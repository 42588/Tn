# 输入与键位待处理

## 定位

本文记录输入法、命令面板搜索和 Windows 快捷键兼容性中仍未解决或待真机确认的问题。

## 背景

Tn 的终端和编辑器需要同时处理 GPUI key event、Windows IME 合成、`EntityInputHandler` 文本投递和自定义快捷键。部分场景 headless 很难验证,必须在真实 Windows + 中文输入法环境中确认。

## 具体问题或计划

### 命令面板中文搜索 parked

- 问题:命令面板目前主要靠 `on_key` 读取 ASCII。IME 合成中文只经 `EntityInputHandler` 投递,命令面板未接该链路,因此只能稳定搜索英文。
- 影响:中文命令、中文文件名或中文动作名称无法直接搜索。
- 计划:命令面板输入框改接与编辑器一致的文本提交路径,把 IME composition 和 committed text 写入搜索 query。

### `Ctrl+Shift+*` 可能被系统或 IME 吞掉

- 问题:中文 / 多布局 Windows 环境中,部分 `Ctrl+Shift+*` 组合可能被输入法或系统输入语言热键拦截。
- 影响:快捷键在开发机可用,但在用户真实环境中不稳定。
- 计划:真机不达时改键,或在文档中要求把系统“输入语言热键”设为未分配。

### `VK_PROCESSKEY` 修复真机待验

- 问题:IME 合成键子类化修复理论上符合 IMM32 标准,但仍需确认微软拼音是否以 `VK_PROCESSKEY` 投递合成键。
- 影响:若判断错误,导航键或合成键仍可能在特定输入法状态下失效。
- 计划:在真实 Windows + 微软拼音中开启 `RUST_LOG=tn::ime=debug`,观察合成键和导航键投递路径。

## 退出条件或验证

- 命令面板能输入中文 query,候选列表实时过滤,Esc/Enter/方向键不被 IME 干扰。
- `Ctrl+Shift+*` 在中文输入法开启、英文输入法开启、多布局切换后均有明确结果:可用、改键或文档化系统设置。
- `RUST_LOG=tn::ime=debug` 能证明微软拼音合成键路径符合预期,且导航键不被 `VK_PROCESSKEY` 子类化吞掉。
