# 工作区与代码包边界

## 定位

本文承接原 `docs/系统架构索引.md` 中 Cargo 工作区、crate 职责、依赖方向、headless 边界和新增 crate 规则。更长的环境与构建命令共享入口见 [工作区与构建约定](../共享记忆/工作区与构建约定.md)。

## 背景

Tn 使用多 crate 工作区,核心理由是把 GPUI pre-1.0 的 API 抖动隔离在 `tn-ui`,让终端内核 headless 可单测,并让 PTY、配置、Agent、编辑器等职责边界清晰。

## 具体内容

### 工作区结构

```text
Tn/
├── Cargo.toml
├── rust-toolchain.toml
├── .cargo/config.toml
├── deny.toml
├── crates/
│   ├── tn-core/
│   ├── tn-pty/
│   ├── tn-config/
│   ├── tn-shell/
│   ├── tn-blocks/
│   ├── tn-editor/
│   ├── tn-agent/
│   ├── tn-ai/
│   ├── tn-ui/
│   ├── tn-app/
│   └── tn-cli/
```

### crate 职责

| Crate | 职责 | 边界 |
|---|---|---|
| `tn-core` | 终端引擎:alacritty 包装、VTE 解析、网格、快照、颜色、滚动、选择、搜索、URL 检测、generation 和 resize 逻辑 | 无 GPUI、无 IO;不把 alacritty 类型泄漏到边界外 |
| `tn-pty` | PTY 后端:`PtyBackend` trait、`LocalPty`、WSL、SSH、远端 SFTP 文件服务和远端命令服务 | 进程/远端连接边界;对 UI 只发结构化事件或同步 Read/Write |
| `tn-config` | 配置与主题:TOML schema、`%APPDATA%\Tn`、默认配置、主题、quick terminal、editor 配置、agent manifest 追加/删除 | 字段全 `#[serde(default)]`;缺失/损坏回退内置默认,不 panic |
| `tn-shell` | shell 集成:旁路 `vte::Parser` 解析 OSC 133/633/7 为 `BlockEvent`,注入 PowerShell 集成脚本 | 原始字节仍照常喂给 `tn-core` |
| `tn-blocks` | Warp 式 block 状态机:Prompt/Input/Running/Finished 到 `Block` | 行锚点来自终端绝对行;agent/alt-screen 拥有 viewport 时隐藏 chrome |
| `tn-editor` | headless 编辑核心:Document、Selection、CursorSet、UndoStack、SearchState、LineLayout 等纯模型 | 无 GPUI;供 Quick Look 和未来 EditorElement 复用 |
| `tn-agent` | Agent Host 平台:开放 `AgentId`、descriptor、capability、runtime、event、registry、usage、external adapter | 对具体 agent 零知识;不认识 Claude/Codex |
| `tn-ai` | 内置 Claude/Codex adapter:解析 JSONL 用量、pricing、pane session detection | 可移除 adapter 层;具体 agent 知识只准在这里 |
| `tn-ui` | GPUI 前端:TerminalView、输入、渲染、文件树、Quick Look、workspace、SSH overlay、Agent chrome、Quick Terminal | 唯一链接 GPUI 的库;消费 headless crate |
| `tn-app` | 二进制 `tn`:开窗、接线、日志、崩溃保护 | 可链接 GPUI |
| `tn-cli` | headless ConPTY 烟雾测试工具和集成探针 | 不依赖 GPUI |

更细的 crate 工作区摘要仍可从 [工作区与构建约定](../共享记忆/工作区与构建约定.md) 进入。

### 依赖方向

```text
tn-app -> tn-ui -> gpui
              -> tn-blocks -> tn-shell -> tn-core
              -> tn-ai -> tn-agent -> tn-config
              -> tn-pty
              -> tn-config
tn-cli -> tn-core + tn-pty
tn-editor 独立 headless
```

约束:

- `gpui` 只能出现在 `tn-ui` / `tn-app`。
- `tn-core`、`tn-pty`、`tn-config`、`tn-agent`、`tn-editor` 必须 headless 可编译、可测试。
- `tn-blocks -> tn-shell -> tn-core` 保持单向。
- `tn-config <- tn-agent <- tn-ai` 保持平台和内置 adapter 分层。
- `AgentRuntimeKind` 是进程/协议运行位置;`FileNamespace` 是文件 I/O 命名空间,两者不能混同。

### 关键 crate 详情

#### `tn-core`

- `GridSize { rows, cols }` 实现 alacritty `Dimensions`。
- `Terminal` 持有 `Term<ChannelListener>`、`Processor` 和事件 `Receiver`。
- 主要 API:`advance(&[u8])`、`resize(GridSize)`、`snapshot() -> TerminalSnapshot`、`drain_events() -> Vec<Event>`。
- `TerminalSnapshot` 包含 rows、cols、cursor、cells,并提供 `rows_text()`、`to_text()` 便于渲染和测试。
- 已落地:每格 fg/bg、ANSI16/256/OSC/INVERSE、Palette、CellRun/row_runs run 批处理、滚动、选择、跨历史搜索、URL 检测和 generation。
- 后续:damage 脏行追踪、滚动区字节上限、styled/has_link/is_prompt/dirty 行提示位、自包含 block 快照。

#### `tn-pty`

- `PtyBackend` 统一 `resize`、`take_reader`、`writer`、`killer`、`wait`、`try_wait`。
- `LocalPty` 经 `portable-pty` 打开 ConPTY 并托管本地 shell;必须保留 slave 句柄。
- WSL 不需要专属 backend,复用 `LocalPty` 跑 `wsl.exe -d <distro> --cd ~`;`wsl.rs` 解码 `wsl -l -q` 的 UTF-16LE。
- `SshBackend` 用 russh 的 async channel,专属 current-thread tokio 线程桥接成同步 Read/Write。
- `tn-pty::LocalPty` 是本地进程环境边界,负责清理 VS Code/Cursor 注入变量,并把 Tn 当前进程 PATH 的额外 shim 目录合并回子 shell PATH。

#### `tn-config`

- 已落地 schema 包括 `[general]`、`[font]`、`[appearance]`、`[quick_terminal]`、`[editor]`、`[[profiles]]`、`[[actions]]`、`[[keybindings]]`、`[[agents]]`。
- 首次启动写默认 `config.toml` 和 `themes/tn-dark.toml`;缺失/损坏整体回退内置 Tn Dark。
- 主题模型包括 `ansi[16]`、`terminal{}`、`ui{}`。
- 手动热重载入口是 `Ctrl+Shift+R`。
- 配置追加工具包括 `append_profile`、`append_agent`、`remove_agent`、TOML fragment 和 block range 逻辑,用于应用内 Agent 编辑器等。
- 后续方向:配色导入、notify 文件监听、防抖 + `arc-swap` 热替换、appearance opacity/backdrop 和 font fallback 应用。

#### `tn-shell` / `tn-blocks`

- alacritty 不解析 OSC 133/633,所以 `tn-shell` 在 PTY 字节上挂旁路 `vte::Parser`,只处理 `osc_dispatch` 并产出 `BlockEvent`。
- PowerShell 集成脚本通过 `-EncodedCommand` 注入,per-session nonce 防伪,`TN_NO_SHELL_INTEGRATION` 可关闭。
- `tn-blocks` 把 `BlockEvent` 归约为命令、cwd、输出行区间、退出码和时长。
- alt-screen 或 agent surface 拥有 viewport 时,block chrome 整体隐藏。

#### `tn-agent` / `tn-ai`

- `tn-agent` 定义 `AgentId`、`AgentDescriptor`、`AgentCapabilities`、`AgentRuntimeKind`、`AgentEvent`、`AgentStatus`、`AgentAdapter`、`GenericAdapter`、`ExternalEventAdapter`、`ExternalProcessAdapter`、`AgentRegistry` 和 `AiUsage`。
- `tn-ai` 保留 Claude/Codex 内置 adapter,解析本地会话 JSONL 和 pricing,并通过泛化的 `resolve_pane_session(&dyn AgentAdapter)` 绑定 pane 会话。
- 详细边界见 [智能体宿主平台](智能体宿主平台.md) 和 [智能体宿主平台化](../新增模块/智能体宿主平台化.md)。

#### `tn-ui`

- `TerminalView` 是终端实体:持有 `Terminal`、writer、backend、focus、grid size、cell width 等状态。
- `workspace` 负责窗口 chrome、标签、n-ary 分屏、命令面板、SSH 连接器、布局管理器、Agent 编辑器等。
- `quick_look` 负责文件/Diff 速览浮层和编辑态;编辑核心已经下沉到 `tn-editor`。
- `explorer` 是文件树侧栏;Workspace 通过 per-pane snapshot 恢复每个 pane 的文件树状态。
- 渲染与 UI 宿主细节见 [终端渲染与界面宿主](终端渲染与界面宿主.md)。

### 新增 crate 规则

1. 新建 `crates/<name>/{Cargo.toml,src/lib.rs}`,字段使用 `*.workspace = true`。
2. 在根 `Cargo.toml` 的 `members` 加入该 crate。
3. 若被其它 crate 复用,在 `[workspace.dependencies]` 加 `tn-<name> = { path = ... }`。
4. 先判断是否必须链接 GPUI;除 `tn-ui` / `tn-app` 外不要引入 GPUI。
5. 维护无环依赖方向,并确保 headless crate 能单测。
6. 改依赖版本走 `[workspace.dependencies]`;git 依赖必须钉死 `rev=`。
