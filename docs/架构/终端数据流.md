# 终端数据流

## 定位

本文承接原 `docs/系统架构索引.md` 中输入、输出、ConPTY、PtyWrite、重绘、resize 和 scrollback 相关数据流。终端窗口尺寸与滚动修复记录见 [终端滚动与窗口尺寸](../修复与优化/终端滚动与窗口尺寸.md)。

## 背景

Tn 的核心链路是双向字节流:前端把键盘编码成终端字节写给 PTY,子进程输出文字和控制序列,`tn-core` 解析为网格快照,`tn-ui` 渲染为像素。

## 具体内容

### 全景图

```text
用户输入                                           子进程输出
   |                                                  ^
   v                                                  |
键盘事件 -> 输入编码 -> PTY writer       PTY reader <- shell / TUI
                     |                   |
                     v                   v
                  ConPTY / SSH channel / WSL
                                     |
                                     v
                         tn-core Terminal::advance
                                     |
                                     v
                         TerminalSnapshot / row_runs
                                     |
                                     v
                              tn-ui / GPUI 渲染
```

本地 shell 经 `LocalPty` 和 ConPTY;WSL 复用 `LocalPty` 跑 `wsl.exe`;SSH 由 `SshBackend` 把 russh async channel 桥成同步 Read/Write。

### 输出路径

1. `LocalPty` 通过 ConPTY 开伪终端,把 shell 挂为子进程。
2. reader 线程循环 `reader.read(buf)`,把字节交给 `Terminal::advance`。
3. `advance` 调用 alacritty `vte::ansi::Processor::advance(&mut Term, byte)`,维护网格、光标和滚动。
4. `Terminal::snapshot()` 遍历可见内容,产出不可变 `TerminalSnapshot`。
5. `TerminalView::render` 把 snapshot 转成 GPUI 可渲染结构,当前路径是 div/text + run 批处理,后续方向是自定义 `TerminalElement`。

### 输入路径

键盘事件经 `TerminalView::on_key_down` 到 `crate::input::encode_key(&Keystroke, InputMode)`,再写入 PTY writer。

已实现的编码规则来自 Windows Terminal `_encodeRegular`:

- 方向键、Home、End 按 DECCKM 选择 CSI 或 SS3。
- 带修饰键时编码为 `ESC[1;<mod><final>`,`<mod>` 为 SHIFT/ALT/CTRL bit + 1。
- F1-F4 支持 SS3/CSI,F5-F20 使用 DECFNK 跳号表。
- Insert、Delete、PgUp、PgDn 使用 `ESC[n~`。
- Backspace 默认为 `0x7f`,Ctrl+Backspace 为 `0x08`。
- Tab 和 Shift-Tab 分别是 tab 与 `ESC[Z`。
- Enter 支持 CR、LNM-CRLF、Ctrl-LF。
- Alt 是 ESC 前缀;CSI 键的 Alt 折进修饰位。
- `Ctrl+Shift+*`、`Ctrl+Tab` 等保留给 UI 时返回 None。

模式位来自 `tn_core::Terminal::input_mode()`,读取 alacritty `Term::mode()`。后置方向包括 kitty 键盘协议、DECKPAM 小键盘编码、win32-input-mode 和 bracketed-paste 包裹。

### PtyWrite 回写

alacritty 解析设备查询时会通过 EventListener 产生 `Event::PtyWrite(reply)`。这些回复必须写回 PTY。

关键原因:ConPTY 启动时可能发 `ESC[6n` 查询光标位置并阻塞等待回复。如果 Tn 只读不回,子进程会卡住,实测表现为只读到少量字节且永不退出。因此 reader 线程在 `advance` 后必须 `drain_events()`,把所有 `PtyWrite` 写回 writer。

另一个 ConPTY 关键点是 EOF 不可靠。判断子进程退出不能只等 `read() == 0`,需要通过 `try_wait` 轮询并在烟雾测试里加硬超时和 watchdog kill。

### 重绘循环

当前策略是 push + vsync 合并:

- reader 线程有新输出时向 `futures::channel::mpsc::unbounded` 发送 wake。
- `dirty` 原子标志去重,通道中最多一个待处理 wake。
- 前台 `cx.spawn` 任务等待 wake 后调用 `cx.notify()`。
- GPUI 把多次 notify 合并到 vsync 帧时钟。
- 空闲时零唤醒,不做轮询。

DEC 2026 同步输出由 alacritty `vte` `Processor` 的 `StdSyncHandler` 处理。网格仅在 BSU -> ESU 完成或超时时变更,所以 `snapshot()` 看到的是整帧,tn-ui 不需要额外半帧防撕裂逻辑。边角情况是同步超时只在下一次 `advance` 时复查,中途卡住的同步流会延迟显示;真实程序通常会及时发 ESU。

### resize 联动

`TerminalView::render` 用 viewport 尺寸、实测等宽字符宽度和行高计算列数/行数。尺寸变化时依次调用:

1. `Terminal::resize_conpty_with_anchoring`
2. `PtyBackend::resize`
3. ConPTY `window_change` 或 SSH `window_change`

尺寸计算走 `TerminalView::fit_grid_size_from_bounds`:

- 隐藏、最小化、标签切换期间,canvas bounds 可能很小但非零。
- 若扣除 `BODY_PAD` 后不足一个完整 cell / row,本帧跳过 resize。
- 这样避免把临时布局写成 `1x1` PTY 尺寸,导致光标回左上并覆盖历史。

grow 锚定策略:

- 普通 shell 用 `ResizeAnchoring::Top`,保住 scrollback 历史。
- agent / alt-screen 用 `ResizeAnchoring::Bottom`,避免把当前 TUI 主屏帧推进 scrollback。

`resize_conpty` 的原因:ConPTY 行/列必须精确跟随 alacritty,否则会复活高频空白。但 alacritty 标准 resize 在长高时底锚定,会把历史行提升进视口顶部;随后 ConPTY 顶锚定 repaint 覆盖这些格,且 alacritty 已把它们移出历史环,造成永久丢失。`resize_conpty` 在 ConPTY 重绘前把被提升的行推回滚动区,让普通 shell 长高表现为顶锚定,与 Windows 控制台一致。

### scrollback 与光标

长期规则:

- `display_offset > 0` 时 snapshot 隐藏 live cursor,避免阅读历史时把真实光标投到历史视图。
- 用户输入前如果 `display_offset > 0`,先回到底部。
- agent pane 与 alt-screen pane 的滚轮交给程序自身,普通 shell 才滚 Tn scrollback。
- 鼠标命中只接受 `BODY_PAD` 内真实网格矩形,恢复帧坐标、padding 和右下空白不 clamp 到有效 cell。

细节修复和单测见 [终端滚动与窗口尺寸](../修复与优化/终端滚动与窗口尺寸.md)。
