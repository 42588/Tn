# 架构总览与关键决策

## 定位

本文承接原 `docs/系统架构索引.md` 中的系统定位、终端基础原理、依赖图、关键技术决策、许可证、已知坑入口和术语表。母文档只保留入口,具体判断依据在本文维护。

## 背景

Tn 是一个 Windows 优先、Rust 编写、GPU 加速的终端模拟器,目标是兼顾性能与视觉体验,并围绕 vibe coding 把 Claude Code / Codex 等 AI CLI 作为一等公民托管。决策背景与多方案权衡曾记录在 `~/.claude/plans/windows-vibe-coding-claude-codex-rust-w-hazy-token.md`。

## 具体内容

### 终端不等于 shell

Shell 和终端模拟器是两个完全不同的层:

| 层 | 是什么 | 在 Tn 里谁负责 |
|---|---|---|
| Shell,例如 PowerShell / cmd / bash | 命令解释器。解析命令、跑程序、管道。它没有 GUI,只读 stdin、写 stdout/stderr。 | 复用现成 shell,不重写 |
| 终端模拟器,例如 Tn / Windows Terminal / iTerm | 屏幕 + 键盘。把 shell 输出的文字和 ANSI/VT 转义码解析成二维字符网格并渲染;把键盘事件编码成字节写回 shell。 | Tn 负责 |

Windows 默认终端曾是 `conhost.exe`。Tn 的职责是用自己的实现取代窗口、输入、解析和渲染层,PowerShell 本身不变,同一套机制也能跑 `cmd`、`wsl`、`bash`、`vim`、`claude` / `codex`。

如果不解析 ANSI/VT,屏幕会直接出现类似 `\x1b[32m...\x1b[0m` 的控制字节。终端的核心工作是把这些序列翻译成光标移动、颜色变化、清屏、alt-screen 等操作,并维护网格、滚动历史和光标状态。Tn 复用 `alacritty_terminal` 的解析引擎,不自研 VT100 全套解析。

### 系统边界

- `tn-core` 维护终端网格、VT 解析、快照、搜索、选择、URL 检测等 headless 能力。
- `tn-pty` 负责本地 ConPTY、WSL 进程托管、SSH backend、远端文件和远端命令服务。
- `tn-ui` 是唯一 GPUI 前端,负责窗口、TerminalView、布局、输入、文件树、Quick Look、Agent chrome、SSH 覆盖卡等。
- `tn-agent` 是对具体 agent 零知识的宿主平台;Claude/Codex 解析在 `tn-ai` 中作为可移除内置 adapter。
- `tn-app` 负责开窗、接线、日志和崩溃保护;`tn-cli` 负责 headless ConPTY 烟雾测试。

### 依赖关系

```text
tn-app -> tn-ui -> gpui
              -> tn-blocks -> tn-shell -> tn-core
              -> tn-ai -> tn-agent -> tn-config
              -> tn-pty
              -> tn-config

tn-cli -> tn-core + tn-pty
```

铁律:

- `gpui` 只能出现在 `tn-ui` / `tn-app`。
- `tn-core` / `tn-pty` / `tn-config` / `tn-agent` / `tn-editor` 等 headless crate 必须能在无 GPU 环境编译和测试。
- `tn-config <- tn-agent <- tn-ai` 是平台到内置 adapter 的单向依赖;平台层不认识 Claude/Codex。
- 不要把 alacritty 内部类型泄漏到 crate 边界外;目前 `Event` 经 `TermEvent` 重导出是唯一例外,需慎用。

crate 详情见 [工作区与代码包边界](工作区与代码包边界.md)。

### 关键技术决策

| 领域 | 决策 | 理由 |
|---|---|---|
| UI / 渲染 | GPUI,Windows 走 DX11 + DirectWrite | 性能与视觉兼顾,纯 Rust,适合文字密集型应用 |
| 引擎 | `alacritty_terminal` | 久经考验的 VT 解析和网格实现,不自研终端控制序列 |
| PTY / 远程 | `portable-pty` 处理 ConPTY/WSL,`russh` 处理 SSH | 纯 Rust、可内置、连接管理可自控 |
| AI 模式 | 托管 CLI + Warp 式 block UI | 不做原生侧栏/内联补全,优先做好 CLI 托管、chrome 装饰和优雅降级 |
| 许可证 | GPL-3.0-or-later | 规避 GPUI 依赖树里 GPL-3.0 传递依赖冲突 |
| MVP 顺序 | 本地内核 -> WSL/SSH -> blocks -> AI | 先有扎实可日用终端 |
| 渲染缓存 | 按 `generation` 缓存渲染数据 | grid 变更才重建 snapshot/row_runs,空闲帧可复用 |
| ConPTY 行数 | 始终精确跟随 alacritty | 尺寸不匹配会复活高频空白;曾试 shell 行锁定已撤销 |
| resize 长高 | 普通 shell 顶锚定;agent / alt-screen 底锚定 | shell 防 ConPTY repaint 覆盖历史;agent/TUI 避免当前主屏推进 scrollback |
| resize 隐藏帧 | 跳过不足一个 cell / row 的临时 bounds | 标签切换 / 最小化恢复的坏 bounds 不能写成 `1x1` PTY 尺寸 |
| scrollback 光标 | `display_offset > 0` 时隐藏 live cursor;输入前回 bottom | 阅读历史时不投影 live cursor 到历史视图 |
| agent 退出 | `-NoExit` 托管 agent 用哨兵标题自报退出 | PowerShell 存活时无 `ProcessExited`;不做脆弱进程树轮询 |

### 依赖与许可证

| Crate | 版本 / 范围 | 许可证 | 用途 |
|---|---|---|---|
| `gpui` | 0.2.2 | Apache-2.0 | UI/GPU 渲染;Windows 使用原生 DX11 + DirectWrite |
| `alacritty_terminal` | 0.26 | Apache-2.0 | Term/Grid/VTE Processor/Selection/damage |
| `portable-pty` | 0.9 | MIT | ConPTY + WSL |
| `russh` / `russh-keys` / `russh-config` | 0.52-0.54 | Apache-2.0 | 纯 Rust SSH |
| `tokio` | 1 | MIT | SSH 后端 async runtime |
| `serde` / `toml` / `directories` / `notify` / `arc-swap` | 工作区统一 | MIT/Apache | 配置、主题、热重载 |
| `tracing` / `tracing-subscriber` | 工作区统一 | MIT | 日志 |

GPUI 在 Windows 上使用 `windows` crate 的 `Direct3D11`、`DirectWrite`、`DirectComposition`、`Dxgi` 特性;blade/wayland/x11/ashpd 等 Linux 相关依赖由 cfg 门控。运行时日志中 `HRESULT(0x887A002D)` 只是可选 DXGI debug layer 缺失,不等于渲染设备失败。

后续可评估 `gpui-component` 获取现成面板、输入框和命令面板组件,但必须先与现有 Calm Glass 规则对齐。

### 已知坑入口

迁移自原母索引的长期坑位:

- ConPTY 启动时的 DSR 查询必须通过 alacritty `PtyWrite` 回复写回 PTY。
- ConPTY 不可靠发 EOF,子进程退出判断要用 `try_wait` 轮询。
- ConPTY 上过早 drop `SlavePty` 会扰动伪控制台,`LocalPty` 必须保留 slave 句柄。
- `CommandBuilder` 默认继承父环境,不是空环境。
- `gpui::Pixels.0` 私有,取值用 `f32::from(pixels)`。
- GPUI async 更新形态是 `cx.spawn` + `WeakEntity::update` + `cx.notify()`;定时器走 `background_executor().timer(dur).await`。
- `cargo` 在 `%USERPROFILE%\.cargo\bin\cargo.exe`,有些 shell 环境 PATH 不一定包含它。

### 术语表

- PTY / ConPTY:伪终端。一对内核管道,让程序以为自己连着真实终端。ConPTY 是 Windows 10+ 的实现。
- VT / ANSI 转义序列:`ESC[...` 这类控制码,表达颜色、光标移动、清屏等。
- OSC:Operating System Command,格式为 `ESC ] ... ST`,用于标题、cwd(OSC 7)、shell 集成(OSC 133/633)、超链接(OSC 8)等。
- DSR:Device Status Report,例如 `ESC[6n` 查询光标位置,终端需回 `ESC[<row>;<col>R`。
- alt-screen:全屏程序如 vim 或 agent TUI 使用的备用屏缓冲,退出后恢复主屏。
- block:Warp 式概念,一条命令及其输出作为可折叠、复制、重跑的单元。
- damage / 脏行:本帧相对上帧变化的行,用于局部重绘。
- snapshot:`tn-core` 产出的不可变可见网格视图,供渲染层消费。
