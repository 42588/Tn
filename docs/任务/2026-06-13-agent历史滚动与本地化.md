# Agent 历史滚动 & 「本地化」澄清

## 背景

BUG 发现清单(③):agent 面板历史「能滚一部分,太长就看不到」;用户记得有个
「历史太多就本地存储、查看时再从本地拉起」的功能,怀疑它坏了。

## 调研结论

1. **滚动本身是好的**。Claude Code 是主屏(main-screen,Ink 渲染),对话进终端回滚缓冲。
   `on_scroll` 里 `scroll_wheel_drives_terminal_app(agent, alt_screen)` 对主屏 agent 返回
   false → 滚轮直接滚终端回滚(`Terminal::scroll`),alacritty 把偏移 clamp 到 history 内。
2. **真因 = 回滚缓冲上限**。`config.general.scrollback_lines` 默认 **5000**,在
   `terminal_view/mod.rs` 的 `Terminal::with_scrollback` 处生效(配置注释「not yet wired」是
   过期的,其实已接线)。长 Claude 会话输出轻松超 5000 行,更早的行被 alacritty 环形缓冲淘汰
   → 滚到顶就到头,「太长就看不到」。
3. **「本地化」功能确实存在,但不是历史扩展器**。`tn-core` 的 `swap_out_async` /
   `restore_if_swapped` 把**后台 tab** 的整块网格序列化到磁盘(`%TEMP%/tn/scrollback_{id}.bin`)
   以释放内存,任何交互(滚动/输出/resize)再从盘拉回。它是**省内存**机制,拉回的是同一块
   (同样 5000 行上限)的网格,**不会**让你看到更早的历史。用户记忆把这个省内存的 swap
   误当成了「历史扩展」。

## 实现

`crates/tn-config/src/config.rs`:

- `General::scrollback_lines` 默认 **5_000 → 50_000**。
- 修正过期注释「(not yet wired)」:说明它已接线、是回滚行数上限、惰性分配(内存随实际输出
  增长而非按上限预分配)、后台 tab 会 swap 到磁盘,所以放大默认是可负担的。

为什么是 50k 而非更大:alacritty 0.26 的 `max_scroll_limit = scrolling_history`,即此值就是
上限、无额外引擎封顶;但它在内存里,marathon 会话产 50k 行满宽输出会吃不少 RAM。50k 覆盖
绝大多数长会话,且惰性分配让常规会话成本不变,后台 tab 又落盘,取了平衡。可在配置里再调大。

## 验证

- `cargo test -p tn-config --lib`:43 绿。
- 真机:跑一个超 5000 行的长 agent 会话,确认能向上滚到更早内容(待用户确认)。

## 后续 / 注记

- **真·无限历史 = 磁盘分页回滚**:alacritty 不支持把超出上限的历史分页到磁盘按需拉回,
  现有 swap 只搬「上限内的整块」。要做到 overnight 会话也不丢,需要自建 disk-paged scrollback,
  属大功能,本次未做。50k 是当前实际可达上限。
- 配置键 `[general] scrollback_lines` 用户可自行调更大(代价是活动 pane 的内存)。
