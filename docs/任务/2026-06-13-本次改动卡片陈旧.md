# 「本次改动」卡片陈旧:列着却点开说没改动

## 现象(真机复现)

活动栏「本次改动 · 2」列出 TODO.md(+1)、config.rs(+9 −2),点 TODO.md 卡片打开 QuickLook
却显示「无改动 · git working tree clean」。这正是 BUG①(增删数字不对 + 点开却说没改动)。

(同图佐证:顶栏 CTX 47% · 474K / 1.0M,context 窗口修复已生效。)

## 根因

活动栏数据来自 `gitutil::changes_for` = `git diff HEAD --numstat`。它由文件变更监听器
(`spawn_change_watcher`)在工作区文件变化时去抖刷新。监听回调用 `is_noise_path` 过滤事件,
而 **`.git` 整个被当噪音过滤**(原意:`.git` 在每次 git 操作——包括我们自己的 diff/status
读取——都会变动,全放行会狂刷/自激)。

副作用:`git commit` / `checkout` / `reset` 这些 **移动 HEAD** 的操作改变了 `git diff HEAD`
的结果(文件变干净),但它们只动 `.git` 内部、不改工作区文件内容 → 没有非 `.git` 事件 →
监听器不刷新 → 卡片停留在提交前的状态。用户点卡片时 QuickLook 现算一遍 diff → 工作区已干净
→「无改动」。「数字不对」同源:卡片是旧快照。

本例的 commit 是 agent 在自己进程里跑 `git commit`,Tn 看不到命令边界,唯一可观测信号就是
文件系统里 `.git/logs/HEAD` 等 ref 状态变化——但被噪音过滤拦掉了。

## 修复

`crates/tn-ui/src/gitutil.rs` 重写 `is_noise_path`:`.git` 内部仍是噪音(index、objects、
locks、COMMIT_EDITMSG…),**但 ref 状态放行**——`.git/` 后第一段是 `HEAD` / `ORIG_HEAD` /
`logs` / `refs` / `packed-refs` 的不算噪音。这些只被 HEAD-moving 操作改写;我们自己的只读
`git diff` / `git status` 从不写它们 → 不会自激刷新循环。

于是 commit/checkout/reset 写 `.git/logs/HEAD`(及 `refs/heads/<branch>`)→ 事件通过过滤 →
去抖后 `refresh_changes` 重算 `git diff HEAD` → 卡片更新到提交后的真实状态。

`is_noise_path` 与 explorer 树监听共用,所以 explorer 的 git 状态标签也会在提交后刷新(同样
是期望行为,且其 `git status` 只读不写 ref → 无循环)。

## 验证

- `cargo test -p tn-ui --lib`:190 全绿,含新增
  `noise_filter_keeps_git_ref_state_but_drops_internals`(.git 内部→噪音、ref 状态→放行、
  工作区文件→放行、target/node_modules→噪音,含绝对路径用例)。
- 真机:提交后活动栏应在 ~300ms 去抖后自动清掉已提交的卡片(待用户重新构建后确认)。
