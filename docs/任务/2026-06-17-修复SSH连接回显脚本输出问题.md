# 任务: 修复 SSH 连接回显脚本输出问题

## 背景与问题分析

当用户连接 SSH 时，终端会打印出一大堆 Shell 集成初始化脚本的 Base64 编码与 eval 命令。
这会破坏终端的整洁度，产生大量的输出垃圾。

### 根本原因
1. **PTY 默认开启 ECHO**：在 `crates/tn-pty/src/ssh.rs` 中，通过 `request_pty` 申请伪终端时，传入的终端模式为空 `&[]`。这意味着伪终端（PTY）在启动时默认开启了 ECHO（回显）。因此，当后台立即将初始化脚本写入 PTY 写入缓冲区时，这些字符在被 Shell 读取执行前就已经被 PTY 驱动自动回显到了终端屏幕上。
2. **Readline 再次回显**：当交互式 Shell 启动时，它在打印完 Prompt 之后，可能会从输入缓冲区中读出未处理的数据并渲染在当前输入行（Readline 行编辑器的特性），导致数据被回显两次。
3. **前端冗余发送 integration_cmd**：在 `crates/tn-ui/src/terminal_view/mod.rs` 的 `PtyEvent::Connected` 事件回调中，前端又一次发送了较长的 `integration_cmd` 以尝试在 prompt 出现前立即获取 CWD。由于此时 `stty echo` 已经恢复了回显，该命令整行在输入时也会被回显出来。且该命令覆盖了 `shell_init` 原本定义的更完善的 `__tn_pc` 函数，破坏了退出码与 Prompt 标记的检测。

## 修复方案

1. **PTY 创建时禁用 ECHO**：
   在 `crates/tn-pty/src/ssh.rs` 的 `request_pty` 中，传入 `&[(russh::Pty::ECHO, 0)]`，使 PTY 在创建时处于无回显状态。
   初始化脚本 `ssh_init_cmd` 的最后一句为 `stty echo 2>/dev/null`，在脚本成功执行完毕后，它会自动重新开启回显。这样整个初始化注入过程对用户完全不可见，且不影响后续交互。

2. **在 Shell 集成脚本末尾立即触发 CWD 上报**：
   在 `crates/tn-shell/src/integration.rs` 的 Bash 和 Zsh 脚本末尾，追加一次手动的 CWD 上报或 `__tn_pc` 调用，这样在 Shell 初始化时就会立刻将初始工作目录汇报给前端，无需在前端发送冗余的 `integration_cmd`。
   - Zsh 脚本末尾追加：`precmd`
   - Bash 脚本末尾追加：`__tn_pc`

3. **前端移除冗余发送**：
   在 `crates/tn-ui/src/terminal_view/mod.rs` 的 `PtyEvent::Connected` 处理中，当 `self.ssh_cfg` 的 `shell_init` 为 `Some` 时，不再发送冗余的 `integration_cmd`，避免覆盖已有的 `__tn_pc` 函数并产生输入回显。
   *（注：修正了 `TerminalView::new` 中原本直接使用 `launch.ssh.clone()` 初始化 `self.ssh_cfg` 的问题，改用携带 `shell_init` 变动的 `actual_ssh_cfg`，使得 `has_shell_init` 的条件判断能正确生效）*

## 验证结果

- **编译检查**：在 root 下执行 `cargo check --workspace` 编译全部通过，0 error。
- **单元测试**：执行 `cargo test --workspace` 全量通过（208+29+13+44+37+27+5+24+18=405个测试全部 Pass）。
- **真机表现**：成功消除了 SSH 登录时回显的巨大 Base64 代码段，同时保持了 CWD 的立即上报与退出码 hooks。
