# 用量环 context 窗口准确(1M 显示成 200K)

## 背景

BUG 发现清单(④第一部分):用量环上下文信息不准确——明明是 1M 模型却显示 200K。

## 根因

Claude 的 session JSONL **不记录** context window 大小(不像 Codex 会记 `model_context_window`),
所以 `tn-agent::pricing::pricing_for` 只能从模型 id 猜:`id.contains("1m") ? 1M : 200K`。
这个启发式对**当前世代**是反的——现代 Claude(Opus 4.6/4.7/4.8、Sonnet 4.6、Fable 5/Mythos 5)
默认就是 **1M**,只有 Haiku 与旧家族是 200K。用户跑 `claude-opus-4-8`(1M),id 里没有字面
"1m",于是错读成 200K;原代码只有在已用量超过 200K 时才回退放宽到 1M。

真机 session 日志确认 id 形如:`claude-opus-4-8` / `claude-sonnet-4-6` / `claude-fable-5` /
`claude-haiku-4-5-20251001`(带日期快照)。

模型目录来源:claude-api 技能参考(缓存 2026-06)。

## 实现

`crates/tn-agent/src/pricing.rs`:

- 新增 `pub fn context_window_for(model: &str) -> u32`:按家族 + 世代子串判定 1M:
  `contains("1m")`(显式长上下文变体)/ `fable` / `mythos` / `opus-4-6` / `opus-4-7` /
  `opus-4-8` / `sonnet-4-6` → 1M;其余(Haiku、旧 Opus/Sonnet、非 Claude、未知)→ 200K。
  用子串 + 世代而非精确 id 表,带日期快照(`…-20251001`)也能命中。
- `pricing_for` 改为调用 `context_window_for`。
- 安全网保留:`claude.rs` / `codex.rs` 仍在「已观测用量 > 窗口」时把 `context_max` 放宽,
  防环读数 >100%(现在主要对 200K 家族起作用)。

测试:
- `pricing.rs::context_window_matches_current_catalog`:现代→1M、Haiku/旧/未知→200K。
- 更新 `claude.rs` 中 `claude-opus-4-7` 相关断言为 1M;`observed_context_above_window_infers_1m`
  改用 Haiku(200K 家族)真正走放宽路径。

## 验证

- `cargo test -p tn-agent -p tn-ai --lib`:tn-agent 15、tn-ai 18 全绿。
- 真机:用户当前 `claude-opus-4-8` 环分母应从 200K 变为 1.0M(待真机确认)。
