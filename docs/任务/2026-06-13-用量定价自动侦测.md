# 用量定价自动侦测(在线价目表 + 内置兜底)

## 背景

BUG④定价部分:`pricing.rs` 里 Opus = $15/$75(旧 Opus 价),当前 Opus 4.6/4.7/4.8 实为
$5/$25,成本读数偏高 3 倍;Fable 5 落到 unknown 显示 $0。用户希望**自动侦测定价**而非手维护表。

## 约束(为什么不能「纯自动」)

`$` 费率不在 Tn 能观测的任何信号里:session 日志只有 token 数(无成本),Models API 只返回
上下文窗口不返回价格,且 Tn 没有自己的 Anthropic 凭据(像 `ccusage` 一样只读本地日志)。
所以定价只能来自**某张表**;「自动」只能是「让这张表能自更新」。用户选定:**在线拉公开
价目表(LiteLLM)+ 内置兜底**。

## 架构

`tn-agent` 是纯无 IO 的 crate(deps 仅 tn-config/serde),不能在它里面发 HTTP。于是分层:

- **tn-agent/pricing.rs(纯)**:
  - `Pricing`(加 serde 以便缓存)。
  - `pricing_builtin`:按家族+世代的内置兜底——Opus 4.6+ → $5/$25/$6.25/$0.50、
    旧 Opus(4.5/4.1/4.0/3)→ $15/$75/$18.75/$1.50、Fable/Mythos → $10/$50/$12.50/$1.00、
    Sonnet $3/$15、Haiku $1/$5、Codex 占位、未知 → 0(不瞎猜)。**离线即修好 Opus bug**。
  - `PricingTable::from_litellm_json`:解析 LiteLLM 的 `model_prices_and_context_window.json`
    (per-token → per-MTok;`max_input_tokens` → 上下文窗口;缺 cache 价按 1.25×/0.1× 推导);
    逐条宽松解析,单条坏行/`sample_spec` 跳过不毁整表。
  - `PricingTable::get`:精确 → 去 `-YYYYMMDD` 日期后缀 → 最长前缀匹配 → miss。
  - 进程级 `RwLock<Option<PricingTable>>` + `set_pricing_table` / `has_loaded_table`(数据,非 IO)。
  - `pricing_for`:先查已加载表,miss 回退 `pricing_builtin`。调用点签名不变。
- **tn-ui/pricing_sync.rs(IO)**:`install_cached` 启动即把磁盘缓存装入表(离线、即时);
  `spawn_refresh` 在开启且缓存过期(>7 天)/缺失时,detached 线程 `reqwest::blocking` 有界 GET
  (8s 超时,像 gitutil)→ 解析 → 装表 → 回写缓存(`%APPDATA%/Tn/pricing_cache.json`)。
  全程 best-effort,失败/禁用就留在缓存/内置兜底。`run()` 启动时调用(无需 cx)。
- **tn-config**:`[general] pricing_auto_refresh`(默认 true)+ `pricing_url`(可改,默认 LiteLLM)。
- **tn-ui/Cargo.toml**:reqwest 加 `blocking` feature。

查上下文窗口也顺带统一:表命中时窗口取自 LiteLLM 的 `max_input_tokens`,miss 回退
`context_window_for`。

## 验证

- `cargo build -p tn-ui` 通过;`cargo test -p tn-config -p tn-agent -p tn-ai -p tn-ui`:
  43 + 18 + 18 + 190 全绿。新增测试覆盖:内置各世代价/缓存价推导、LiteLLM 解析+日期后缀+
  前缀匹配、加载表覆盖内置再清空、context 目录核对。
- **首次启动**:无缓存 → 立即用内置兜底(当前模型已正确)→ 后台拉取数秒后填入完整表。
