# 性能分析报告

> 生成日期:2026-06-22 · 方法:全量源码静态走查 + 现有计装(`TN_PERF`)+ 现有 `criterion` bench 通道。
> 本报告**不含**真机大规模 profiling 数据(需用户配合采样),但给出每个热点的可复现测量入口。
> 母索引见 [质量保障索引](README.md)。

## 1. 范围与方法

- 走查对象:`crates/` 下 11 个 crate,约 56K 行 Rust(其中 `tn-ui` 占 42.5K)。
- 关注三类成本:**每帧 CPU**(渲染重建)、**前台线程阻塞**(同步 spawn / 同步 IO)、**内存与分配 churn**。
- 判定口径:以「空闲零成本、忙时有界、最坏路径不阻塞前台」为合格线。
- 局限:GPUI 的 GPU paint 阶段耗时、DirectX 提交延迟无法纯静态测量,本报告标注为「需探针」。

## 3. 热点路径逐项分析

### 3.1 终端引擎:`snapshot()` + `row_runs()` 每次新输出全量重建 ★关键热点

- **路径**:每次 paint 若 `generation` 变化 → [snapshot()](../../crates/tn-core/src/lib.rs#L987) 分配 `Vec<SnapshotCell>`(rows×cols)→ [row_runs()](../../crates/tn-core/src/lib.rs#L371) **再**重建一遍 `Vec<Vec<SnapshotCell>>` 网格,然后按样式合并 run 并为每段 `SmolStr::new` 分配。
- **复杂度**:O(rows×cols),典型 40×120 = 4800 cell;**同一帧两次全网格分配**(snapshot 一次、row_runs 内 `grid` 一次)。
- **现有缓解**:generation 缓存使其只在「有新输出」时触发,光标闪烁/失焦不触发。
- **残留风险**:整屏高频重绘类 agent(Claude Code/Ink、codex)**每帧都 bump generation**,于是每帧吃满这条全量重建——这正是 `Cargo.toml:84` 注释点名的「§2.1 全量重建」开发期卡顿来源。
- **已有 bench**:[benches/engine.rs](../../crates/tn-core/benches/engine.rs) 覆盖 `advance_200_colored_lines` 与 `snapshot_and_row_runs_40x120`。
- **优化方向(按收益/风险)**:
  1. ✅ **已落地(2026-06-22)**:`row_runs` 把 `Vec<Vec<SnapshotCell>>`(`self.rows + 1` 次堆分配)改为**单块扁平 `rows*cols` 缓冲**,scatter 与逐行扫描在同一块上完成,语义不变。实测 `snapshot_and_row_runs_40x120` **76.3 µs → 71.4 µs(-5.7%,criterion p<0.05「improved」)**,每次重建分配数 `rows+1 → 1`。见 [lib.rs:row_runs](../../crates/tn-core/src/lib.rs#L371)。
  2. (待办)`snapshot()` 复用持久化 `cells` 缓冲——受渲染缓存保护,仅在「有新输出」时分配,收益次于第 1 项;若做需改 `&self`→内部可变或调用方持有缓冲。
  3. (长期 P3)行级脏标记(只重建变化的行),把全量重建降为增量——收益最大但改动面大,需谨慎对待 alacritty `display_iter` 语义。

### 3.2 会话启动在前台线程同步阻塞 ★关键缺陷(即本轮用户提问的「进入很慢」)

- **路径**:磁贴点击 [launch_in_active_tab](../../crates/tn-ui/src/workspace.rs#L1381) 与命令面板 Enter [launch_profile_in_tab](../../crates/tn-ui/src/workspace.rs#L2222) 均汇流到 [spawn_pane_with → install_pane](../../crates/tn-ui/src/workspace.rs#L1773),在 GPUI 事件回调(前台线程)内**同步**执行 [TerminalView::new](../../crates/tn-ui/src/terminal_view/mod.rs#L893)。
- **阻塞成本**:
  - 本地:`LocalPty::spawn` = `openpty()`(创建 ConPTY 伪控制台内核对象)+ 子进程 `CreateProcess`([local.rs:24](../../crates/tn-pty/src/local.rs#L24))。
  - SSH:[SshBackend::spawn](../../crates/tn-ui/src/terminal_view/mod.rs#L920) 阻塞至网络握手完成——**最坏路径,可达秒级前台冻结**。
- **影响**:点磁贴 / 敲命令进入工具时窗口短暂冻结;SSH 尤其明显。注意这是**前台阻塞**,与 GPUI 渲染性能无关。
- **优化方向**:把 spawn 移到后台执行器,前台先插入「启动中…」占位 pane,backend 就绪后回填;SSH 握手天然适合异步化(`russh` 本就是 async)。这能直接消除点击冻结。

### 3.3 宠物 30fps CPU 栅格化:动画期持续分配 churn

- **路径**:动画+可见时,每 33ms [render_rgba](../../crates/tn-ui/src/pet_lottie.rs#L212) 分配 `vec![0f32; pw*ph*4]` 累积缓冲 → 转 RGBA8 → `RgbaImage::from_raw` → `RenderImage::new` → 上一帧 `cx.drop_image` 驱逐([pet.rs:1351](../../crates/tn-ui/src/pet.rs#L1351))。
- **评估**:精灵尺寸小,**绝对成本低**;且 fingerprint 去重保证静帧零重绘、隐藏即停。属于「可接受但有改进空间」。
- **残留**:动画播放期间每帧三次堆分配(f32 buffer、RGBA buffer、RenderImage)+ 一次 GPU 纹理驱逐,构成稳定的分配 churn。
- **优化方向**:① 复用 f32/RGBA 缓冲(`Vec` 重用);② 对循环段按「量化 frame × breed × scale」做栅格缓存(循环动画帧数有限,可命中复用),把动画期降为查表。优先级低于 3.1/3.2。

### 3.4 次要观察

- **explorer 重建**:`FilesChanged` 触发 [explorer.rebuild](../../crates/tn-ui/src/workspace.rs#L1788) 全量重建目录树 + git 标记;大仓库 + 高频文件事件下可能成为成本点。当前未见去抖合并(SSH 路径已是命令结束后才刷,本地走文件 watcher)。**建议**:确认本地 watcher 是否对 burst 事件去抖;大仓可加目录树惰性展开。
- **reader → advance 锁内执行**:已用 inner `catch_unwind` 防 mutex 中毒([io.rs:127](../../crates/tn-ui/src/terminal_view/io.rs#L127)),设计正确;16 KiB 缓冲是吞吐与锁延迟的权衡点,若未来出现「超大输出吞吐不足」可作为可调参数。
- **多 pane 叠加**:每个 pane 各自持有 reader 线程 + repaint task + blink task + exit watcher([mod.rs:1028-1046](../../crates/tn-ui/src/terminal_view/mod.rs#L1028));pane 数量很多时线程/任务数线性增长,但单 pane 空闲零成本,通常可接受。

## 4. 内存

- **分配器**:mimalloc 静态链接为 `#[global_allocator]`,配空闲感知 `mi_collect`(见 §2)。设计良好:忙时不回收,空闲段只回收一次。
- **QuickLook**:图片走 `RenderImage` 旁路 + SIMD 缩放消除 ~400MB 临时峰值;PDF LRU(MAX=8 页)+ 渲染限高 + 延迟 GC,解决 ~200MB 常驻(见 [2026-06-16 任务](../任务/2026-06-16-QuickLook内存与GPU占用优化.md))。
- **宠物纹理**:每帧驱逐上一帧 GPU 纹理,无显存累积。
- **潜在点**:`row_runs` 的 `SmolStr` 每段分配——短 run 内联(≤22 字节)不堆分配,长 run 会堆分配;整屏彩色长行场景下分配量随 run 数增长(与 3.1 同源)。

## 8. 一句话结论

整体性能纪律良好(空闲零成本体系完整);**唯一用户可直接感知的问题是会话启动同步阻塞前台(P0-1,需真机验证,尚未实施)**,其次是终端整屏重绘类 agent 下的全量重建分配(P1)。详细补测见 [单元测试计划](单元测试计划.md)。
