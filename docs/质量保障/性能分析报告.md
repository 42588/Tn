# 性能分析报告

> 生成日期:2026-06-22 · 方法:全量源码静态走查 + 现有计装(`TN_PERF`)+ 现有 `criterion` bench 通道。
> 本报告**不含**真机大规模 profiling 数据(需用户配合采样),但给出每个热点的可复现测量入口。
> 母索引见 [质量保障索引](README.md)。

## 1. 范围与方法

- 走查对象:`crates/` 下 11 个 crate,约 56K 行 Rust(其中 `tn-ui` 占 42.5K)。
- 关注三类成本:**每帧 CPU**(渲染重建)、**前台线程阻塞**(同步 spawn / 同步 IO)、**内存与分配 churn**。
- 判定口径:以「空闲零成本、忙时有界、最坏路径不阻塞前台」为合格线。
- 局限:GPUI 的 GPU paint 阶段耗时、DirectX 提交延迟无法纯静态测量,本报告标注为「需探针」。

## 3. 热点路径逐项分析

### 3.1 终端引擎:`snapshot()` + `row_runs()` 每次新输出全量重建 ★关键热点

- **路径**:每次 paint 若 `generation` 变化 → [snapshot()](../../crates/tn-core/src/lib.rs#L987) 分配 `Vec<SnapshotCell>`(rows×cols)→ [row_runs()](../../crates/tn-core/src/lib.rs#L371) **再**重建一遍 `Vec<Vec<SnapshotCell>>` 网格,然后按样式合并 run 并为每段 `SmolStr::new` 分配。
- **复杂度**:O(rows×cols),典型 40×120 = 4800 cell;**同一帧两次全网格分配**(snapshot 一次、row_runs 内 `grid` 一次)。
- **现有缓解**:generation 缓存使其只在「有新输出」时触发,光标闪烁/失焦不触发。
- **残留风险**:整屏高频重绘类 agent(Claude Code/Ink、codex)**每帧都 bump generation**,于是每帧吃满这条全量重建——这正是 `Cargo.toml:84` 注释点名的「§2.1 全量重建」开发期卡顿来源。
- **已有 bench**:[benches/engine.rs](../../crates/tn-core/benches/engine.rs) 覆盖 `advance_200_colored_lines` 与 `snapshot_and_row_runs_40x120`。
- **优化方向(按收益/风险)**:
  1. ✅ **已落地(2026-06-22)**:`row_runs` 把 `Vec<Vec<SnapshotCell>>`(`self.rows + 1` 次堆分配)改为**单块扁平 `rows*cols` 缓冲**,scatter 与逐行扫描在同一块上完成,语义不变。实测 `snapshot_and_row_runs_40x120` **76.3 µs → 71.4 µs(-5.7%,criterion p<0.05「improved」)**,每次重建分配数 `rows+1 → 1`。见 [lib.rs:row_runs](../../crates/tn-core/src/lib.rs#L371)。
  2. ✅ **已落地(2026-06-22,P1-2)**:与其「复用缓冲」,直接把 `snapshot()`+`row_runs()` 两步合并为 [`Terminal::render_frame()`](../../crates/tn-core/src/lib.rs#L999) 单遍——一块扁平 `rows*cols` 网格**直接**从 `renderable_content` 生成,省掉中间 `Vec<SnapshotCell>` 与二次 scatter(每次重建分配减半)。UI 渲染路径已切到它,与旧路径**逐字节相同**(等价性单测 `render_frame_matches_snapshot_then_row_runs` 兜底;`resolve_fg_bg`/`build_runs` 抽出共享,杜绝两路漂移)。实测 `render_frame_40x120` **66.8µs** vs 旧 `snapshot_and_row_runs` **83.8µs**(同机同轮 **≈-20%**)。
  3. (长期 P3-1)行级脏标记(只重建变化的行)——**重评后降级**:alacritty 虽有 line damage,但整屏重绘类 agent(Claude Code/Ink、codex)每帧 damage 几乎所有行,对真正的热路径收益有限;而单行变化场景本就便宜。高改动面 + 低边际收益 + 无法离屏验证 → 暂不做。

### 3.2 会话启动:前台阻塞 vs 子进程启动延迟(即本轮用户提问的「进入很慢」)★需先实测

> ⚠️ **本节较初版有重大更正**:经核读 backend 源码,SSH 的 spawn **不阻塞前台**;「很慢」的主因是子进程启动延迟而非前台冻结。详见下。

- **路径**:磁贴点击 [launch_in_active_tab](../../crates/tn-ui/src/workspace.rs#L1381) 与命令面板 Enter [launch_profile_in_tab](../../crates/tn-ui/src/workspace.rs#L2222) 均汇流到 [spawn_pane_with → install_pane](../../crates/tn-ui/src/workspace.rs#L1773),在 GPUI 事件回调(前台线程)内**同步**执行 [TerminalView::new](../../crates/tn-ui/src/terminal_view/mod.rs#L893)。
- **前台同步成本(真正阻塞点击的部分)**:仅本地 `LocalPty::spawn` = `openpty()`(创建 ConPTY)+ 子进程 `spawn_command`(`CreateProcess`)+ clone reader([local.rs:24-89](../../crates/tn-pty/src/local.rs#L24))。这些是**返回即结束**的 OS 调用,**不等待子进程做任何事**;量级通常为数十 ms(**待实测**,见下计装)。
- **SSH 不阻塞前台(更正)**:[SshBackend::spawn](../../crates/tn-pty/src/ssh.rs#L303) 文档明确「Start the session thread. **Returns immediately**; connect/auth happen on the [session thread]」——它只起一个带 tokio runtime 的专用线程,连接/认证/握手全在后台 async 完成并经事件回吐(`russh` 本就是 async)。**初版「SSH 阻塞至握手完成、可达秒级前台冻结」属误判,此处更正。**
- **「很慢」的真正主因**:子 shell 打出**第一个 prompt 前**的启动延迟——pwsh 加载 profile + 模块 + 注入的 `-EncodedCommand` 集成脚本;agent 则是 node 进程启动。这发生在 spawn 返回**之后**,是子进程自身耗时,**异步**到达:pane 已显示但空白,直到 prompt 字节抵达。**async-spawn 不能缩短这一段。**
- **计装 + 真机实测(2026-06-22)**:[TerminalView::new](../../crates/tn-ui/src/terminal_view/mod.rs#L893) 在 `tn::spawn` target 记录前台 spawn 耗时。真机日志(NVIDIA RTX 4060,debug 构建):

  | program | elapsed_ms(前台同步) | 用户感知 |
  |---|---|---|
  | powershell.exe(首次冷启) | **296** | — |
  | powershell.exe(后续) | **5–18** | 基本无延迟 ✓ |
  | wsl.exe | **5–7** | **~3s** ✗ |
  | agent(codex/claude,托管于 pwsh) | **5–18** | **~5s** ✗ |

  **诊断坐实**:前台 spawn 全部 ≤18ms(首次 296ms 是 ConPTY / 侧载 conpty.dll 一次性初始化);而用户对 WSL/agent 感知到 3–5s。三者走同一 `LocalPty::spawn`,**pwsh 无延迟即证明前台不是瓶颈**——那 3–5s 完全是**子进程自身启动**(`wsl.exe` 拉起发行版;agent 的 node 进程 + 初始化),发生在 spawn 返回**之后**、异步到达。
- **P0-1 定论:async-spawn 无效,不做。** 它只能省下 ≤18ms 的前台时间(不可感知),动不了 3–5s 的子进程启动;且 UI 本就没冻结(前台 5ms 就返回了)。正确方向是 **UX 缓解**:在 pane 收到首字节前显示「启动中…」占位,消除「点了没反应」的观感。子进程启动本身提速属 codex/wsl/node 范畴,Tn 不宜改其行为。

### 3.3 宠物 30fps CPU 栅格化:动画期持续分配 churn

- **路径**:动画+可见时,每 33ms [render_rgba](../../crates/tn-ui/src/pet_lottie.rs#L212) 分配 `vec![0f32; pw*ph*4]` 累积缓冲 → 转 RGBA8 → `RgbaImage::from_raw` → `RenderImage::new` → 上一帧 `cx.drop_image` 驱逐([pet.rs:1351](../../crates/tn-ui/src/pet.rs#L1351))。
- **评估**:精灵尺寸小,**绝对成本低**;且 fingerprint 去重保证静帧零重绘、隐藏即停。属于「可接受但有改进空间」。
- **残留**:动画播放期间每帧三次堆分配(f32 buffer、RGBA buffer、RenderImage)+ 一次 GPU 纹理驱逐,构成稳定的分配 churn。
- **优化方向**:① ✅ **已落地(2026-06-22,P2-1)**:复用 f32 累积缓冲——`render_rgba` 拆出 [`render_rgba_into(scratch)`](../../crates/tn-ui/src/pet_lottie.rs#L213),由 `PetView.lottie_scratch` 持有并复用(`clear`+`resize` 置零),免去动画期每帧 `pw*ph*4` floats(16 字节/px,比输出大 4×)的堆分配;输出 `out` 仍须新分配(交给 `RenderImage` 持有,不可复用)。等价性单测 `render_rgba_into_reuses_scratch_byte_identically`(脏/暖 scratch 均逐字节一致)兜底。② (可选增强)对循环段按「量化 frame × breed × scale」做 `RenderImage` 缓存,把稳态循环降为查表——收益更大但 key 正确性需 GUI 验证,留作后续。

### 3.4 次要观察

- **explorer 重建**(P2-2,**核查后确认无虞**):文件 watcher 已是 **500ms trailing-edge 去抖**——每个新事件重置静默计时,停下后才刷一次([explorer.rs:628](../../crates/tn-ui/src/explorer.rs#L628));并经 `is_noise_path` 挡掉构建产物;且 [rebuild()](../../crates/tn-ui/src/explorer.rs#L951) 在**后台线程**走目录、用 `rebuild_rev` 守卫丢弃过期结果,不阻塞前台。报告初版「未见去抖合并」的担心不成立。大仓进一步优化(惰性展开)可留作可选增强。
- **reader → advance 锁内执行**:已用 inner `catch_unwind` 防 mutex 中毒([io.rs:127](../../crates/tn-ui/src/terminal_view/io.rs#L127)),设计正确;16 KiB 缓冲是吞吐与锁延迟的权衡点,若未来出现「超大输出吞吐不足」可作为可调参数。
- **多 pane 叠加**:每个 pane 各自持有 reader 线程 + repaint task + blink task + exit watcher([mod.rs:1028-1046](../../crates/tn-ui/src/terminal_view/mod.rs#L1028));pane 数量很多时线程/任务数线性增长,但单 pane 空闲零成本,通常可接受。

## 4. 内存

- **分配器**:mimalloc 静态链接为 `#[global_allocator]`,配空闲感知 `mi_collect`(见 §2)。设计良好:忙时不回收,空闲段只回收一次。
- **QuickLook**:图片走 `RenderImage` 旁路 + SIMD 缩放消除 ~400MB 临时峰值;PDF LRU(MAX=8 页)+ 渲染限高 + 延迟 GC,解决 ~200MB 常驻(见 [2026-06-16 任务](../任务/2026-06-16-QuickLook内存与GPU占用优化.md))。
- **宠物纹理**:每帧驱逐上一帧 GPU 纹理,无显存累积。
- **潜在点**:`row_runs` 的 `SmolStr` 每段分配——短 run 内联(≤22 字节)不堆分配,长 run 会堆分配;整屏彩色长行场景下分配量随 run 数增长(与 3.1 同源)。

## 7. 如何复现测量(测量入口)

```powershell

$env:TN_PERF = "1"; cargo run -p tn-app
#   → %APPDATA%\Tn\logs\tn.log 内 target=tn::perf 行

# 2) 终端引擎热点基线
cargo bench -p tn-core            # advance / snapshot_and_row_runs

# 3) 会话 spawn 前台耗时(已内置计装,target=tn::spawn)

#    → 日志行的 elapsed_ms = TerminalView::new 前台阻塞耗时,分别看 本地 pwsh / WSL / SSH / agent
```

## 8. 一句话结论

整体性能纪律良好(空闲零成本体系完整);**用户可感的「进入很慢」主因是子 shell 首 prompt 的异步启动延迟(非前台冻结)——前台同步部分仅 `openpty`+`CreateProcess`,SSH 更是立即返回;已加 `tn::spawn` 计装待实测(P0-1)**。其次是终端整屏重绘类 agent 下的全量重建分配(P1,§3.1.1 已优化 -5.7%)。详细补测见 [单元测试计划](单元测试计划.md)。
