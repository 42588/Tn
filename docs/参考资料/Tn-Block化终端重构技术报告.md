# Tn: Block 化终端重构可行性与技术报告

## 1. 背景与目标

自上世纪 70 年代以来，终端界面一直延续着“纯文本字符流（Text Stream）”的形式。所有的命令输入、程序输出、报错信息都被杂糅在一块无限滚动的画布上。这种缺乏结构化的形态导致了开发者在现代开发中面临诸多痛点：
- 难以精准选中、复制或分享某一次特定命令的长输出。
- 难以直接从视觉上区分上一个命令和下一个命令的边界。
- AI 介入时，难以提取精确的“当前上下文”。

受现代化终端项目（如 Warp）的启发，本报告旨在探讨并在工程上论证：**将 Tn 项目当前的纯文本流终端，重构为基于“语义块（Block-based）”的现代架构的可行性与实现路径。**

---

## 2. 核心理念：何为 Block？

Block 化架构的核心在于**将“一次交互”封装为一个独立的结构单元**。

一个完整的 Block 包含：
1. **输入区 (Command)**：用户输入的命令内容。
2. **输出区 (Output)**：该命令产生的全部 stdout / stderr 文本流。
3. **元数据 (Metadata)**：执行时刻 (Timestamp)、工作目录 (CWD)、耗时 (Duration)、退出状态码 (Exit Code)。

通过在数据结构上划分 Block，界面 UI 即可将其渲染为独立的组件，从而解锁折叠、一键复制、精确定位、以及基于单体 Block 的 AI 分析等现代交互能力。

---

## 3. 技术实现拆解

基于 Tn 项目当前的技术栈（Rust + GPUI + Alacritty/portable-pty），实现 Block 化主要分为三个核心技术层：

### 3.1 数据源层：Shell 注入与语义识别 (Semantic Prompts)
要切分连续的字符流，不能依赖正则匹配（因为输出随时可能与 prompt 相似），必须依赖 Shell 本身发出的明确信号。业界标准方案是利用 **OSC 133 语义提示转义序列**。

我们需要在 Tn 启动时，向宿主 Shell（Bash / Zsh / PowerShell）注入一段启动脚本（类似 iTerm2 shell integration），让 Shell 在执行生命周期中打印如下控制符：
- `OSC 133 ; A ST`：提示符（Prompt）开始
- `OSC 133 ; B ST`：命令输入（Command）开始
- `OSC 133 ; C ST`：命令执行（Execution）开始，即按下回车
- `OSC 133 ; D ; <ExitCode> ST`：命令执行结束，带回退出码

### 3.2 解析层：PTY 拦截与状态机
在 `crates/tn-app/` 的终端解析循环中，我们需要扩展现有的 Alacritty VT 解析器：
1. **拦截器**：拦截并消耗掉 `OSC 133` 指令，不让其进入常规的网格渲染。
2. **状态机映射**：
   - 当收到 `A` 到 `C`，我们将输入的字符捕获为 `CurrentCommand`。
   - 当收到 `C`，触发一个事件：`BlockStarted`。
   - 随后的字符流统统路由给当前活跃的终端网格（Terminal Grid）实例。
   - 当收到 `D`，触发事件：`BlockFinished(ExitCode)`，封卷当前的终端网格，并为下一次输入准备新的终端实例。

### 3.3 UI 渲染层：基于“磷光 (Phosphor)”设计语言的 GPUI 重构
目前我们的终端是一个单体的大 Canvas。在 Block 架构下，终端主视图将被重构为一个 GPUI 的 `ListView` 或动态容器：
- **历史流**：每个历史 Block 是一个只读的 `BlockView`。
- **视觉风格**：摒弃常规的单调线条，采用 Tn 专属的**“磷光”设计系统**。活跃的 Block 使用标志性的取景器角标 `[ ]` 聚焦，光标保持唯一的生命色 `#5BE7C4`，并在视觉上完美兼容 Tn 的“幽灵白底” (Ghost Surface) 或深色磨砂材质层次。
- **工具栏**：每个 `BlockView` 包含状态指示、折叠、一键复制等基于上下文的工具链。

## 4. 与 Tn 现有生态的深度化学反应

Block 化绝不仅是排版的改变，它将彻底激活并放大 Tn 现有的那些独一无二的基础设施：

### 4.1 QuickLook 原生预览的无缝接入
Tn 已经实现了 Markdown 预览态与编辑态的秒切功能。在 Block 架构下：
- 当识别到某个 Block 的输出包含 Markdown、JSON 或表格数据时，Block 内部可以直接呈现 `👁 QuickLook` 按钮。一键将枯燥的终端字符流渲染为 **GPUI 原生的富文本界面**，彻底打破传统终端“只能看纯文本”的桎梏。

### 4.2 透明的 AI 账单与 Token 跟踪
基于我们新落地的“用量额度与定价自动侦测”系统：
- **Block 级用量挂载**：每一个由 AI 生成、或调用 Agent 修复的 Block，其元数据栏都可以直接显示本次请求消耗的 Tokens 和具体金额（如 `$0.0012`），让所有 AI 调用在终端粒度上完全透明化。
- **精准隔离除错**：当命令报错时，点击 `✨ Ask AI`，系统只需将**该 Block 的局域输出**及 CWD 喂给内置大模型，极大降低噪音干扰和无谓的 Token 开销。

### 4.3 空间化的宠物陪伴引擎 (v2.3)
Tn 独家的小狗宠物系统将获得全新的“物理空间”：
- **具象的落脚点**：由于界面从“一整块无定形文本”变成了“层叠的实体 Block”，每一个 Block 的横边都能成为宠物的物理平台。当一个红色报错 Block 出现时，小狗甚至可以跳上这个 Block 的边缘进行疑惑地“嗅探”。
- **工作流共情**：遇到耗时极长的构建 Block 时，小狗会在该 Block 旁卧倒打盹；一旦执行完成（不论成败），伴随清脆的“叮”声，小狗会立刻兴奋起身迎接你。

### 4.4 拖拽系统与输入流
结合 Tn 已有的文件拖拽能力：
- 用户可以将本地文件直接拖入活动中的 Block 输入区。得益于结构化的输入框，不仅仅是补全路径，该 Block 还可以把这个文件记录为**上下文资产**，以便随后的 AI Agent 读取并理解执行意图。

## 5. 潜在挑战与风险

1. **Alternate Screen (备用屏幕) 的处理**：
   当用户运行 `vim`、`htop`、`lazygit` 等 TUI 程序时，终端会切换到备用屏幕，这属于“全屏接管”。对于这种情况，Block 模式必须自动退化，将其作为一个特殊的“全屏沉浸 Block”，退出后再恢复列表流。
2. **内存与渲染性能**：
   GPUI 如果同时维护数百个独立的终端网格实例可能会导致内存飙升。需要实现**虚拟列表渲染（Virtualization）**，不在视口内的历史 Block 应冻结终端状态并转化为轻量级的只读文本视图或卸载。
3. **环境兼容性**：
   Windows 下的 PowerShell 历史版本对 OSC 133 支持薄弱，可能需要通过 `ConPTY` 或 `PSReadLine` 的特殊 Hook 来注入语义信号。

---

## 6. 演进路线图建议 (Roadmap)

如果决定启动该计划，建议分三期（Phase）推进，以降低风险：

- **Phase 1: 幽灵注入与数据采集** (对用户无感)
  只实现 Shell 注入和解析器拦截。在后台记录命令开始与结束的边界数据，验证不同系统（Windows/WSL/SSH）下的捕获准确率。
- **Phase 2: UI 解耦与基础 Block 渲染**
  将 GPUI 渲染层切换为 Block 列表，实现视觉分隔、折叠和一键复制功能。处理 Alternate Screen 退化逻辑。
- **Phase 3: 生态赋能**
  绑定 AI Error Fix 工作流，以及打通宠物状态感知。

---

**结论**：将 Tn 重构为基于 Block 的终端，在工程上具有明确的可行性。这将极大拉开 Tn 与传统终端（如 Windows Terminal、Alacritty）的代差体验，是实现下一代 Agentic 终端的核心基础设施。
