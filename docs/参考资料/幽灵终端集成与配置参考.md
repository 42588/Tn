## OSC 133 / 7 / 8

Ghostty 的 OSC 133 语义适合作为 Tn block 引擎的依据:

| 标记 | 含义 |
|---|---|
| `A` | 新提示符开始 |
| `B` | 提示符结束,用户输入开始 |
| `C` | 输入结束,命令执行开始 |
| `D [;exit_code]` | 命令结束,可带退出码 |
| `P;k=i|r|c|s` | 显式提示符起点和种类 |

Tn 的 Warp block 可建模为从提示符与命令输入到命令结束的跨度。解析必须宽容,未知或畸形选项静默忽略。

相关 OSC:

- OSC 7 表示 cwd,payload 是 `file://host/path`;解百分号编码后可用于标签标题、同目录开新标签和 block cwd 元数据。
- OSC 8 表示超链接;空 URI 表示链接结束。URI 驻留后渲染下划线并处理点击。

## shell 集成注入

Ghostty 的 Unix shell 注入机制可借鉴原则,但不逐字照搬:

- bash 通过 `ENV` 和 posix 启动技巧加载集成,再重放正常启动序列。
- zsh 通过临时 `ZDOTDIR` 加载集成后恢复用户配置。
- fish / elvish / nushell 通过前插资源目录加载集成,之后清理环境。
- 安全原则包括:特性开关、幂等、可重入、命令加 `builtin` 防 alias 劫持、多行提示符正确标注、每条命令只发一次结束标记。

Windows / PowerShell 不能照搬 Unix 机制。Tn 应通过覆写 `prompt` 函数和 `PSConsoleHostReadLine`,经 `$PROFILE` 或 `-NoExit -Command` 注入,发出相同 OSC 133 标记。

## PageList 思想

Ghostty 的 PageList 使用固定大小 page 双向链表、页内偏移、样式驻留和行级可能位,并按字节封顶滚动区。

Tn 不重写 PageList,因为 alacritty_terminal 已有网格、滚动区和 reflow。但可以吸收三点:

- 在行数上限之外增加滚动区字节上限,避免长跑 AI / dev 输出和超宽行爆内存。
- 每行维护 styled、has_link、is_prompt、dirty 等“可能”位,假阳性可接受,用于 block 命中测试和重绘 damage。
- block 快照做成自包含、样式驻留的拥有式拷贝,不要指向活网格。

## IO 线程与重绘合并

Ghostty 和 Windows Terminal 在 IO 架构上有相同经验:

- 使用专用阻塞 reader 线程。
- 每块输出整体喂解析器,不要逐字节驱动 UI。
- 有数据就通知,由帧时钟或 vsync 合并渲染。
- 不可见时跳过渲染。
- resize 合并,避免拖拽时高频抖动。
- DEC 同步输出 BSU/ESU 期间抑制重绘,但要有安全复位机制。

Tn 取舍:

- 采纳 push notify + GPUI vsync 合并。
- 避免 8ms dirty 轮询。
- 若 profiling 发现冗余帧,再加最小间隔定时器。

## 配置模型

Ghostty 配置经验:

- 单一类型化结构。
- repeatable 字段表达多值。
- 迁移表处理旧 key。
- diff-on-reload 只应用变化字段。
- 部分字段标注仅对新 surface 生效。
- 手动重载优先,避免文件监听风暴。

结合 Windows Terminal 的设置模型,Tn 采用:

- profile / color scheme / actions / keybindings 解耦。
- font 和 appearance 作为嵌套对象。
- 字段全默认值,便于继承覆盖。
- 可选 notify 文件监听必须防抖。

## 判定

- OSC 133/7/8 语义:采纳。
- shell 注入原则:采纳。
- PowerShell 注入机制:改造。
- PageList 整套内存模型:跳过,只吸收字节上限、行可能位和拥有式快照。
- IO push + vsync 合并:采纳。
- 配置 diff-on-reload 和迁移表:采纳。
