# 终端项目互联网调研报告

## 定位

本文记录 2026-06-10 对终端相关项目的互联网调研结果。范围覆盖终端模拟器、终端工作区、CLI/TUI 生产力工具、AI 编码终端和用量监控工具,用于支持 Tn 后续重新设计。

## 调研口径

- 信息来源优先级:官方官网、官方文档、官方 GitHub 仓库、项目 README。
- 用户偏好不是问卷结论,而是基于公开信号的推断,包括项目定位、README 强调点、功能取舍、安装/使用路径、stars/社区热度等。
- stars、安装量、官网宣传数字会变化,本文只把它们作为弱信号,不作为精确统计。
- OpenClaude 的“终端随机宠物”线索未找到可稳定核验的一手来源,在本文中作为用户偏好线索单独处理,不写成已确认功能事实。

## 总体结论

终端产品的竞争点已经从“能跑 shell”扩展为四层能力:

1. 终端核心:低延迟渲染、字体/IME/颜色/图片协议、跨平台一致性、PTY 生命周期稳定。
2. 工作区:tabs、splits、布局保存、远端连接、profile、会话恢复、多路复用。
3. 生产力:统一 fuzzy picker、结构化历史、Git/TUI 面板、资源监控、命令输出块。
4. AI 与状态:repo-aware agent、可审计 diff、权限模式、后台任务、用量窗口、配额风险提示。

对 Tn 最重要的启发是:不要把 AI 做成覆盖终端的聊天框,而应把它做成终端工作区里的可审计任务层。命令、输出、文件变更、用量、权限、远端位置都应该结构化,这样才能同时服务传统终端用户和 AI agent 用户。

## 用户偏好归纳

| 偏好 | 公开信号 | 对 Tn 的含义 |
|---|---|---|
| 开箱即用,但能深度配置 | Ghostty、Starship、Zellij 都强调默认可用;WezTerm、kitty、Oh My Posh 提供深配置 | 默认界面必须安静可用,高级配置和快捷键逐步暴露 |

| 键盘优先但可视化 | fzf、lazygit、btop、Zellij 说明 TUI 化工作流仍很强 | Tn 应提供命令面板、picker、可预览列表和键盘可达的布局操作 |

| AI 行为需要可审计 | aider、OpenCode、Claude Code、Gemini CLI 都强调文件编辑、diff、运行命令或结构化输出 | AI 改动必须可 diff、可撤销、可复跑验证,并明确权限模式 |

| 终端也需要情绪反馈 | OpenClaude 随机宠物线索虽未核验,但与 mascots、状态条、主题偏好一致 | 可探索低干扰“陪伴/状态反馈”层,默认可关闭,不影响专业感 |

## 用户指定线索整理

### OpenClaude 的终端随机宠物

本轮检索了 OpenClaude 官网、GitHub 仓库以及 “OpenClaude terminal random pet / mascot / 随机宠物” 等关键词。已核验到的 OpenClaude 一手资料只确认其 terminal-first、model-neutral、工具调用、profile 和流式 agent 特性;没有找到“随机宠物”功能的稳定官方来源。

处理结论:

- 不把“随机宠物”写成 OpenClaude 的已确认功能。
- 将其作为用户偏好的弱信号:终端用户可能喜欢轻量陪伴、状态反馈、情绪化小组件和低压提示。
- 如果 Tn 做类似能力,应满足三个约束:默认可关闭、不能遮挡命令输出、必须绑定有用状态,例如 agent 空闲/运行/等待权限/额度告警,而不是纯装饰。

### `ai-limit` 的用量信息能力

`ai-limit` 的价值非常直接:把 Claude Code 和 Codex 的用量、配额、token 消耗和窗口重置时间从隐性状态变成显性状态。它同时提供 CLI 和 GUI/菜单栏,说明“用量监控”既是一次性查询,也是适合常驻展示的环境状态。

对 Tn 的可采用设计:

- 在状态栏显示当前 agent/provider 的额度风险,如正常、接近上限、已限流、未知。
- 提供详情面板展示窗口范围、剩余额度、重置时间、最近消耗、数据来源和刷新时间。
- 明确数据采集来源和副作用,特别是第三方接口、cookie、账户状态、本地日志解析等。
- 用量信息应与任务调度关联:接近上限时建议降级模型、暂停后台 agent、切换 provider 或延后长任务。

## 对 Tn 的设计建议

### 优先级 1: 终端基础可信

- 保持输入、滚动、resize、IME、字体、颜色、复制粘贴和 PTY 生命周期稳定。
- 兼容现有生态:tmux、Zellij、Starship、Oh My Posh、fzf、SSH、WSL。
- 提供极简模式,让重度终端用户可以只使用高质量终端核心。

### 优先级 2: 结构化工作区

- 把命令执行结果分块,记录命令、cwd、exit code、耗时、输出摘要和关联文件。
- 建立统一 picker,覆盖文件、目录、历史、命令、会话、Git 分支、agent 任务。
- 支持 workspace/layout 保存和恢复,尤其是终端、日志、编辑器、预览、agent 状态的组合。

### 优先级 3: AI 任务层可审计

- agent 模式区分只读计划、可编辑执行、后台长任务、管道分析。
- 所有文件改动必须有 diff、来源任务、验证命令和撤销路径。
- shell 命令执行要有权限边界、cwd、环境、输出和失败原因。
- 非交互模式应输出 JSON 或事件流,供 CI、任务面板和外部工具消费。

### 优先级 4: 用量和成本可见

- 内置 provider/account 用量面板,至少支持已接入 agent 的额度状态。
- 额度状态不应只是日志,应进入调度策略和 UI 提示。
- 对未知数据要诚实标注“不可获取/来源不可靠/刷新失败”。

### 优先级 5: 低干扰情绪反馈

- 宠物/mascot/状态角色可以探索,但定位应是“状态反馈”而不是“装饰遮罩”。
- 适合承载 agent 当前状态:空闲、思考、运行命令、等待确认、错误、额度告警。
- 默认不应占据输入焦点,不应改变命令输出布局,不应影响复制和屏幕阅读。

## 可落地功能候选

| 功能候选 | 来源启发 | 建议优先级 | 验收口径 |
|---|---|---|---|
| 结构化命令块 | Warp、Atuin、Claude Code | 高 | 每条命令有 cwd、exit code、耗时、输出摘要和复制入口 |
| 统一 picker | fzf、zoxide、lazygit | 高 | 一个入口搜索历史、文件、会话、任务、Git 对象 |
| Workspace/layout 保存 | WezTerm、Zellij、Windows Terminal | 高 | 可保存和恢复终端/日志/编辑/预览布局 |
| Agent 权限模式 | OpenCode、Claude Code、Gemini CLI | 高 | 只读计划、编辑执行、后台任务权限清晰 |
| 用量监控面板 | `ai-limit` | 高 | 显示额度窗口、重置时间、风险状态和数据来源 |
| 管道式 AI 命令 | Mods、Gemini CLI | 中 | 日志/diff/test output 可直接进入 AI 分析,支持结构化输出 |
| 终端状态宠物 | OpenClaude 用户线索、mascot 偏好 | 低到中 | 默认关闭或低占用;只承载有用状态;不遮挡输出 |
| 插件/扩展 API | Tabby、kitty、Zellij、Crush | 中 | 先稳定核心事件和命令接口,再开放插件 |

## 风险与边界

- 不要以 AI 功能牺牲终端核心性能。终端输入延迟和输出稳定性是底线。
- 不要把用户偏好推断当成定量结论。本文没有做用户访谈或问卷。
- 不要盲目复制竞品功能。Tn 应优先结合现有 GPUI、PTY、agent host、quick look 和 usage 能力。
- 不要把用量数据做成黑盒。用户必须知道数据从哪里来、是否准确、是否触发网络或账户副作用。
- 不要把宠物体验做成强制装饰。它只有在表达状态、降低焦虑或提高可理解性时才有产品价值。
