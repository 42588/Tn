本文是外部源码研读责任域的母索引,只负责说明参考来源、入口、采纳原则、当前状态和新增信息落点。

## 参考来源

- Windows Terminal:ConPTY 的官方实现。主要参考输入编码、AtlasEngine GPU 文字渲染、ConPTY 操作纪律和设置模型。
- Ghostty:高性能终端实现。主要参考 OSC 133 / shell 集成、屏幕与滚动区内存模型、IO 线程与重绘合并、配置热重载。
- 本地浅克隆位置:`d:\coder\_refs\terminal` 与 `d:\coder\_refs\ghostty`。文内源码路径均相对各自仓库根目录。

## 采纳原则

- **采纳**:可直接迁移的终端协议语义、状态机、操作纪律和配置结构。
- **改造**:思想有价值但实现栈不同的设计,例如 AtlasEngine 的批处理思想迁移到 GPUI Element,而不是重写 D3D。
- **跳过**:与 Tn 现有依赖重复、会引入大量 unsafe 或不适合 Rust/GPUI/serde/TOML 栈的机制。
- 参考资料只记录外部项目怎么做、Tn 为什么采用或不采用。Tn 当前事实和路线图最终以 [系统架构索引](系统架构索引.md) 为准。
- 涉及许可证的内容必须记录来源和兼容性;可复制脚本时保留版权声明和署名。

## 状态

- Windows Terminal 的 legacy 输入编码规则已经被 Tn 输入层采纳。
- ConPTY 的 DSR / PtyWrite、EOF、resize 和关闭纪律已进入架构记忆与实现约束。
- Ghostty 的 OSC 133/7/8 语义、shell 集成原则、IO push + vsync 合并、配置 diff-on-reload 模型已进入路线图或已实现。
- AtlasEngine 的渲染结构只作为自定义 GPUI `TerminalElement` 的数据组织参考,不直接复制 D3D 渲染器。
