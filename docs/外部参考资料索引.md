# 取经:从 Windows Terminal 与 Ghostty 吸收的设计

> 本文是对两个标杆终端源码的精读提炼,**只保留对 Tn 当前/规划架构可落地的设计要点**,并给出明确取舍。
> 判定图例:**采纳(Adopt)** 直接照搬设计 · **改造(Adapt)** 借鉴思想、换实现 · **跳过(Skip)** 不适合我们的栈。
> 参考源码已浅克隆到 `d:\coder\_refs\terminal`(C++)与 `d:\coder\_refs\ghostty`(Zig),文件引用均相对各自根目录。
> 总体设计与现状见 [系统架构索引.md](系统架构索引.md)。

为什么选这两个:
- **Windows Terminal** —— ConPTY 的发明者,微软官方实现。最有价值的是**按键→VT 编码**、**AtlasEngine GPU 文字渲染**、**ConPTY 操作纪律**、**设置/配色模型**。
- **Ghostty** —— 以极致性能与一流 shell 集成著称。最有价值的是 **OSC 133 / shell 集成**、**分页屏幕内存模型的思想**、**IO 线程与重绘合并**、**配置热重载**。

## 一、输入:按键 → VT 字节 ⭐(最高价值)

我们现在的 `keystroke_to_bytes`([crates/tn-ui/src/terminal_view.rs](../crates/tn-ui/src/terminal_view.rs))是简化版。**正确完整的实现应照搬 Windows Terminal 的 `_encodeRegular` 算法**(`src/terminal/input/terminalInput.cpp:823-1267`)。

**核心规则**:
- **模式影响编码**:`DECCKM`(光标键应用模式)、`DECKPAM`(小键盘应用模式)、`LNM`(换行模式)、`BackarrowKey`(退格键)、`SendC1`。这些模式位**从 alacritty 的 `Term::mode()` 取**(`APP_CURSOR / APP_KEYPAD / LINE_FEED_NEW_LINE` 等),不要自己维护。
- **修饰键参数 = `mod + 1`**:位 `SHIFT=1, ALT=2, CTRL=4`,合成后**写到线上要 +1**(Ctrl=4→`5`)。仅在非零且该键支持时才发。
- **四种输出形态**:`CSI <num>;<mod> <final>` / `SS3 <final>`(无修饰且对应应用模式开时) / 字面串 / Alt 前缀(ANSI 模式下 Alt → 前置 `ESC`)。

**逐键表(`to_esc_str` 重写依据)**:

| 键 | 规则 |
|---|---|
| 方向键 ←↑→↓(final `D A C B`) | 有修饰**或** DECCKM 关 → `ESC[A`;否则 SS3 `ESC O A`;带修饰 `ESC[1;<mod>A` |
| Home/End(`H F`) | 同方向键的 CSI/SS3 逻辑 |
| F1–F4(`P Q R S`) | 无修饰 SS3 `ESC O P`;带修饰 `ESC[1;<mod>P` |
| F5–F20 | DECFNK `ESC[<n>~`,n 取自 LUT `{15,17,18,19,20,21,23,24,25,26,28,29,31,32,33,34}`(注意有跳号) |
| Insert/Delete/PgUp/PgDn | `ESC[2~ / 3~ / 5~ / 6~`,修饰作 `;<mod>` |
| Backspace | 默认 `0x7f`;BackarrowKey 模式发 `\b`;**Ctrl 反转两者**;Alt 前缀 |
| Tab | `\t`;**Shift+Tab → `ESC[Z`**(CBT) |
| Enter | `\r`(LNM 模式 `\r\n`);**Ctrl+Enter → `\n`**;Alt 前缀 |
| Ctrl+字母/符号 | `_makeCtrlChar`:`@`..`~`→`ch & 0x1f`;Space→`0x00`;`/`→`0x1f`;`?`→`0x7f`;`2`..`8`→`{0,27,28,29,30,31,127}` |
| 可打印字符 | 直接发 GPUI `Keystroke.key_char`(已含 shift/布局);Alt → 前置 `ESC` |

**Ghostty 的结构同样值得借鉴**(`src/input/key_encode.zig`):**单一 `encode(key_event, options)` 入口**,`options` 从终端模式派生;kitty 键盘协议用一张 PUA 码点表(`input/kitty.zig`),其表可作数据直接移植。

**落地**:M1 在 tn-ui/tn-core 边界做独立 input 层。先做 legacy 编码(覆盖 DECCKM/keypad/ctrl/alt),kitty 键盘协议按模式门控后置(alacritty 也支持)。
**判定:采纳** WT 的 regular 算法 + Ghostty 的 `encode(options)` 结构。**跳过** win32-input-mode。

## 二、GPU 文字渲染 ⭐

**Windows Terminal AtlasEngine**(`src/renderer/atlas/BackendD3D.cpp` + `common.h`)的核心思想:**不要"画文字",而是把整屏拍平成一串带类型的 quad,用一次 draw call + 一个会分支的 shader 画出来**。
- **统一 16 字节紧凑 `QuadInstance`**(`shadingType + position + size + texcoord + color`),所有字形/背景填充/光标/下划线/选区都进同一个实例缓冲,`DrawIndexedInstanced` 一把梭。`shadingType` 枚举告诉 shader 这块是灰度文字 / ClearType / 彩色 emoji 直采 / 实线 / 虚线 / 光标 / 填充。
- **字形图集缓存**:一张会按 2 的幂增长的纹理,用矩形装箱(stb_rect_pack)打包;缓存键 `(fontFace) → (lineRendition) → glyphId`;空白字形跳过。**字形用 DirectWrite/D2D 光栅化进图集**(手写 alpha 纹理路径反而慢)。
- **每行 shape 一次**:`MapCharacters`(字体回退分段)→ `AnalyzeScript/GetGlyphs/GetGlyphPlacements`,产出 SoA 的 `ShapedRow`(glyphIndices/advances/offsets/colors),缓存复用。
- **颜色**:fg/bg 分两张平面位图;彩色 emoji 标 `TextPassthrough` 直采 RGBA;连字(overhang)用 `overlapSplit` 把一个宽字形按 cell 边界切成多个 quad 以便逐格上色。

**Ghostty** 的可移植原则:**按 run(同字体同样式连续段)整形,而非按 cell**(连字/CJK/性能的关键);**box-drawing/Powerline 等用内部 sprite 自绘**,不信字体字形(像素级对齐、无缝隙)。

**重要纠偏**:**GPUI 自带文字系统与字形图集(etagere 装箱)+ DirectWrite**。所以我们**不重写 D3D 渲染器**,而是写一个自定义 GPUI `Element`,在其中:① 把可见网格拍平成 GPUI 的 `paint_quad`(背景/光标/选区)+ `shape_line`/`paint_glyph`(字形)批量提交;② 按 run 整形;③ fg/bg 分离上色;④ box/Powerline 用 quad 自绘。AtlasEngine 的数据结构(typed-quad、ShapedRow、shading 分类、color plane)指导我们**怎么在 GPUI 内组织批处理**。
**判定:改造**(思想移植,API 用 GPUI)。**落地**:M1 的自定义 `TerminalElement`。彩色 emoji 直采、双高(DECDHL)后置。

## 三、ConPTY 操作纪律(Windows Terminal)

`portable-pty` 已封装 ConPTY,但下列**运行规则**必须遵守(`src/cascadia/TerminalConnection/ConptyConnection.cpp`、`src/winconpty/winconpty.cpp`):
1. **EOF 只有两种**:`ReadFile` 读到 **0 字节**,或 **`ERROR_BROKEN_PIPE`**。没有哨兵。(印证我们用 `try_wait` 而非等 `read()==0` 的做法。)
2. **DSR 阻塞是设计如此**:开 `INHERIT_CURSOR` 时 ConPTY 立刻发 `ESC[6n` 并**在收到 `ESC[<row>;<col>R` 回复前不处理任何输入**(`winconpty.cpp:453-460`)。必须把 alacritty 的回写(`PtyWrite` 事件)接到 PTY writer —— **这正是我们卡住过的 bug**(已修,见 [系统架构索引 §3.4](系统架构索引.md))。
3. **流水线化读循环**:在处理上一块之前就把下一个 `ReadFile` 排队,打字延迟不被解析拖累;UTF-8→宽字符用**跨读取保持状态的解码器**(多字节序列可能被读取边界切断),坏字节吞掉不致命。
4. **resize 传参顺序是「列在前、行在后」,都 clamp ≥1**(0 维度 = `E_INVALIDARG`)。先 `Term::resize` 再通知 ConPTY。
5. **关闭**:`CancelIoEx` + join reader 线程后再拆回调,避免 use-after-free。
6. **清屏**:有 `ConptyClearPseudoConsole` 信号(清 PTY 侧滚动区),实现"clear buffer"时用得上。
**判定:采纳** 这些纪律。**落地**:tn-pty。

## 四、屏幕/滚动区与快照(Ghostty PageList)

Ghostty 把整屏+滚动区做成**固定大小 page 的双向链表**(`PageList.zig`/`page.zig`):每页是 mmap 来的页对齐分配,内部一律用 **u32 偏移而非真指针**(故可整页 memcpy/序列化/重定位);cell 是 packed `u64`;样式/超链接/grapheme 在页内**驻留(interning)+ 引用计数**,cell 只存 16 位样式 id;**每行带 styled/grapheme/hyperlink "可能"位**(假阳性可接受),让擦除/滚动快路径整行跳过侧表(自称 4× 提速);**滚动区按字节封顶**,满了**复用最老页的内存**而非逐行丢弃;**pin**(稳定游标)在裁剪/reflow 时自动更新。

**纠偏**:**alacritty_terminal 已拥有网格、滚动区、reflow**(`Grid<Cell>` = 行的 `VecDeque`,按行数封顶)。我们**不重写 PageList**(在 Rust 里重做会巨大、充满 unsafe、且重复)。但吸收三点:
1. **滚动区加字节上限**(不止行数上限)——长跑 dev/AI 输出 + 超宽行易爆内存。包装 alacritty 网格,估算字节,超预算时裁历史。
2. **每行"可能"位**(styled/has_link/is_prompt/dirty,假阳性 OK)——用于 block 命中测试与重绘 damage 的整行跳过。
3. **块快照 = 自包含、样式驻留的拥有式拷贝**(`Vec<Cell>` + 驻留样式表),而非指向活网格的指针 —— 便宜、可序列化、与 reflow 无关。这正是 Warp block 冻结一条命令输出所需的性质。
**判定:跳过**重写 / **改造**思想。**落地**:tn-core(字节上限/快照),tn-blocks(块快照,M3)。

## 五、OSC 133 / 7 / 8 与 shell 集成 ⭐⭐(tn-shell / tn-blocks 的核心,M3)

**Ghostty 的 OSC 133 语义**(`src/terminal/osc.zig` 的 `semantic_prompt.zig`)—— 直接按此规范实现我们的 block 引擎:

| 标记 | 含义 |
|---|---|
| `A` | 新提示符开始(fresh-line)。FTCS_PROMPT |
| `B` | 提示符结束、**用户输入开始**(光标在此打字) |
| `C` | 输入结束、**命令执行开始**(命令输出起点) |
| `D [;exit_code]` | **命令结束**,可带退出码 |
| `P;k=i\|r\|c\|s` | 显式提示符起点 + 种类(initial/right/continuation/secondary),用在 PS1 里避免重绘重复触发 |
| 选项 | `aid=`(shell id,辨别嵌套 shell)、`cl=`(点击移光标能力)、`click_events`(shell 处理鼠标点击)、`redraw=last`。**解析必须宽容:未知/畸形选项静默忽略** |

一个 **Warp block = 从 `A`/`B`(提示符+键入命令)经 `C`(命令开始)到 `D`(退出码)的跨度**。Tn 做法:在字节流里旁路拦截 OSC 133(挂 alacritty 的 OSC Handler),维护状态机 `A/N→开块、抓提示符`;`C→标记命令起始行、冻结命令文本`;`D→关块、记录退出码+时长`。**块边界存成行标记/pin**(借鉴 §四 的 pin),滚动区裁剪与 reflow 后仍有效。
- **OSC 7**(cwd):payload 是 `file://host/path`,**不校验**,解百分号编码后当 cwd → 标签标题、同目录开新标签、每块 cwd 元数据。
- **OSC 8**(超链接):`ESC]8;id=<id>;<uri>ST`,空 URI = 链接结束;URI 驻留、cell 标 link-id;渲染下划线 + 处理点击。

**shell 集成自动注入机制**(`termio/shell_integration.zig`,Ghostty 的精华):
- **bash**:`bash --posix` + `ENV=<res>/bash/ghostty.bash`;脚本检测注入标志后**手动重放** bash 正常启动序列(`/etc/profile`、`~/.bashrc`),再退出 posix 模式并清理自身环境变量;遇 `-c`/`--posix` 直接放弃(不冒险破坏)。
- **zsh**:改 `ZDOTDIR` 指向资源目录(存旧值),其 `.zshrc` 加载集成后还原 `ZDOTDIR` 再 source 用户 zshrc。
- **fish/elvish/nushell**:把资源目录**前插** `XDG_DATA_DIRS`,加载后从中移除以免泄漏给子进程。
- **安全原则**(务必移植):env 特性开关 `GHOSTTY_SHELL_FEATURES=...`、所有命令加 `builtin` 前缀(防用户 alias 劫持)、幂等/可重入(检测是否已装)、`133;D` 每命令只发一次、多行提示符次行用 `133;P;k=s`。

**Windows / PowerShell 现实**:bash/zsh/fish 的 `ENV`/`ZDOTDIR`/`XDG` 技巧**不适用 pwsh**。pwsh 的正确钩子是**覆写 `prompt` 函数 + `PSConsoleHostReadLine`**,经 `$PROFILE` 或 `-NoExit -Command` 注入,**发出相同的 OSC 133 标记**。移植**原则**(幂等、保存/恢复、特性开关、相同标记),不照搬 Unix 机制。bash/zsh/fish 脚本可在 WSL/SSH 落地时**近乎原样复用**。
- **许可证**:Ghostty 的集成脚本是 **GPLv3**(源自 Kitty)。**Tn 本身是 GPL-3.0-or-later → 复用兼容**,保留版权声明/署名即可(OSC 序列语义本身不受版权保护)。
**判定:采纳** OSC 133/7/8 解析与语义 + 注入原则;**改造** 注入机制到 PowerShell;bash/zsh/fish 脚本后续复用。**落地**:tn-shell + tn-blocks(M3),tn-ui(OSC 7 标题、OSC 8 链接)。

## 六、IO 线程与重绘合并(Ghostty + Windows Terminal)

两家一致的架构,印证并细化我们的设计:
- **专用阻塞 reader 线程**:紧循环读到 `WouldBlock`,小缓冲即可(Ghostty 用 1KB,靠紧循环;WT 用 128KB + 流水线)。每块**先拿渲染状态锁 → 整片喂解析器(不逐字节)→ 触发一次合并的重绘通知**。
- **同步输出(DEC 2026)**:BSU/ESU 之间抑制重绘,**但要配 1 秒安全复位定时器**,免得 buggy app 永久冻屏;resize 时强制清。
- **resize 合并 ~25ms**:窗口拖拽时不抖动。
- **渲染节流交给 vsync**:Ghostty 的经验是"别过度工程化批渲染"——有数据就通知,让帧时钟去合并;不可见时跳过渲染。

**纠偏我们现状**:M0 用的是 8ms `dirty` 轮询。**M1 改为:reader 线程数据到达 → `cx.notify()`(push),由 GPUI 帧时钟/vsync 合并**;仅当 profiling 发现冗余帧时再加最小间隔定时器。叠加同步输出 + 1s 复位 + resize 合并。这与 系统架构索引 路线图 M1 的"push + 4ms 合并"一致,这里补充"优先靠 vsync、加同步输出复位"。
**判定:采纳**。**落地**:tn-pty(reader)、tn-core(锁内解析、同步输出)、tn-ui(vsync 合并)。

## 七、配置与主题(Windows Terminal + Ghostty)

**Windows Terminal 的设置模型**(`src/cascadia/TerminalSettingsModel/MTSMSettings.h` + `defaults.json`)值得照搬结构:
- **profile**:`{name, commandline, starting_directory, color_scheme, font{face,size,weight,features,...}, appearance{cursor_shape,...}, history_size(默认9001), close_on_exit(always/graceful/never/automatic), scrollbar, antialiasing, padding}`。**font 与 appearance 是嵌套对象**。
- **color_scheme**:`{name, foreground, background, cursor_color, selection_background, + 16 个 ANSI 名(black..white + bright*)}`;profile 按名引用 scheme,可覆盖 fg/bg/cursor。颜色 `#RRGGBB`。
- **键位解耦**:`actions[]`(`{id, command}`,command 可带参)与 `keybindings[]`(`{keys, id}`)**分两张表** —— 一个动作可绑多键、用户按 id 改键。
- **级联默认**:内置 `defaults.json` ← 用户 `settings.json` ← 动态 profile,每个字段独立可继承覆盖。

**Ghostty 的配置经验**(`src/config/Config.zig`):
- 单一**类型化结构体** + 多值用 `Repeatable*`(对应我们 TOML 里的 list/map)。
- **迁移表**:`old_key → handler`,旧配置跨版本不炸(早加便宜,后补痛苦)。
- **diff-on-reload**:只重新应用变化的字段,部分字段标注"仅对新 surface 生效",避免为改个颜色就重建 GPU/pty。
- **手动重载优先**(keybind 触发),而非文件监听 —— 避免编辑器写临时文件引发重载风暴。

**落地(tn-config,M1)**:serde + TOML,字段全 `#[serde(default)]`(等价 X-macro 的可继承);schema 按 WT 的 profile/scheme/actions+keybindings 结构;加迁移表 + diff-on-reload;**默认手动重载命令,可选 `notify` 文件监听**(带防抖,缓解风暴)。
**判定:采纳**结构与模式。

## 八、明确"不抄"的东西

| 不抄 | 原因 |
|---|---|
| Ghostty PageList 分页内存 / 偏移指针 / 自管 mmap | alacritty 已拥有网格+滚动区+reflow;在 Rust 重做巨大且 unsafe 重 |
| 字体引擎(Collection/HarfBuzz/Atlas/SharedGrid) | GPUI + DirectWrite 已做发现/回退/整形/光栅化;只留"按 run 整形"和"box sprite 自绘"两条原则 |
| 自造 VT/CSI/DCS 解析器(`stream.zig`/`Parser.zig`) | alacritty 的 `vte` 已做;我们只在其上加 OSC 拦截层 |
| libxev 事件循环 / WT 的 WinRT+X-macro 机制 | 用 Rust 线程+channel+GPUI executor/帧时钟代替;serde 结构体代替 X-macro |
| Ghostty 自有配置行格式 | 用 TOML/serde,只留结构模式 |
| Kitty 键盘协议、彩色 emoji 直采、双高字符(初期) | 后置;先把 legacy 编码与单色/彩色 cell 做扎实 |

## 九、对路线图的具体影响(对 系统架构索引 的修订)

- **M1 · 输入层**:按本文§一,用 WT `_encodeRegular` 算法重写 `to_esc_str`;模式位取自 alacritty `Term::mode()`。kitty 协议门控后置。
- **M1 · 渲染**:自定义 GPUI `TerminalElement`,按§二的 typed-quad 批处理 + 按 run 整形 + fg/bg 分离 + box/Powerline sprite 自绘(不重写 D3D)。
- **M1 · 重绘**:把 8ms 轮询改为 push `notify` + vsync 合并 + 同步输出(2026)及 1s 复位 + resize 合并 ~25ms(§六)。
- **M1 · 配置**:tn-config schema 按§七(profile/scheme/actions+keybindings 解耦、嵌套 font/appearance、迁移表、diff-on-reload、手动重载优先)。
- **tn-core**:滚动区加**字节上限**;行加 **styled/has_link/is_prompt/dirty 提示位**(§四)。
- **M3 · tn-shell/tn-blocks**:按§五的 OSC 133/7/8 规范实现;PowerShell 用 `prompt`+`PSConsoleHostReadLine` 注入发相同标记;bash/zsh/fish 脚本(GPLv3,与我们 GPL 兼容)在 WSL/SSH 复用并署名。
- **tn-pty**:落实§三的 ConPTY 纪律(EOF 规则、流水线读、有状态 UTF-8、resize 列在前 clamp≥1、clear 信号、关闭先 cancel+join)。
