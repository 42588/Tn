# 焦点与输入法坑位

## 定位

本文记录 GPUI 焦点、keymap、浮层、IME、中文输入和快捷键相关的长期坑位原则。具体 bug 根因和验证日志写入修复与已知问题专题。

## 坑位原则

### IME 与文本输入

- Windows 上中文合成文本经 `EntityInputHandler` 投递;没有 handler 时只能收到 ASCII `key_char`。
- 接了 `EntityInputHandler` 仍不够:可打印键不能在 `on_key` 中 `stop_propagation`,否则 gpui 不会 `TranslateMessage`,IME 合成不会启动。
- handler 需要在 paint 阶段通过 `window.handle_input` 注册;不要在普通 render body 中直接注册。
- 命名键和带修饰键仍应由 `on_key` 编码或处理,避免双写和终端控制键丢失。
- 微软拼音等 IME 可能不发 `GCS_COMPSTR`,只在提交时发结果串;需要尊重 `VK_PROCESSKEY` 或平台级 IME 归属,不能纯靠 `is_composing`。

### 浮层焦点与事件归属

- 不占分屏的浮层要有 click-away scrim 和面板根事件吞噬,避免点击穿透到底层终端。
- `track_focus` 浮层要在 render 中聚焦,动作回调时浮层可能还没渲染。
- 当前选中 pane 应从 gpui 实际焦点反推,不要只靠鼠标点击回调维护。
- 持自己焦点句柄的浮层必须同时进入“reveal 聚焦块”和 `overlay_focused` 排除列表;漏掉后者会每帧被 workspace 根抢焦点。
- 无 `EntityInputHandler` 的纯导航浮层要进入 IME disabled 集合,否则活动中文输入法可能吞掉方向键 / Enter / Esc。

### Quick Look / 编辑器焦点协作

- 当前活跃实体应独占焦点并自己处理键;跨实体动作通过事件回 workspace。
- Quick Look 关闭后根据入口还焦点给文件树或终端,不要无脑 refocus active pane。
- 预览态和编辑态的文本真源不同;退出编辑时要把 dirty 缓冲镜像回预览数据,但不能写盘或清 dirty。
- dirty-close 守卫必须同时覆盖 app 内关闭动作和 OS 标题栏关闭 / Alt+F4。
- 虚拟列表中光标行高必须固定;鼠标点位和拖选应依赖已知行号,避免读取不可用滚动偏移。

### keymap 和系统快捷键

- `WindowKind::PopUp` 内 keymap/action 可能不派发;Quick Terminal 这类窗口不要依赖主窗口 action 机制完成关键流程。
- `Ctrl+Shift+*` 在中文/多布局 Windows 上可能被系统输入语言热键吞掉;需要允许改键或提示用户关闭系统布局切换热键。
