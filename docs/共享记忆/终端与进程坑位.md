# 终端与进程坑位

## 定位

本文记录 ConPTY、PTY 生命周期、agent 进程托管、SSH/WSL 命名空间和终端 resize 相关的长期坑位原则。

## 坑位原则

### ConPTY 与 PTY 生命周期

- ConPTY DSR 回复必须把 alacritty `Event::PtyWrite` 写回 PTY writer,否则子进程可能卡在启动期查询。
- ConPTY 无可靠 EOF,用 `try_wait` 轮询子进程状态,不要把 `read()==0` 当作唯一退出信号。
- `LocalPty` drop 必须杀子进程;关闭 pane / tab 要能沿视图 drop 链释放 PTY。
- reader 线程持锁推进 VT 解析时要在锁内捕获 panic,避免 Mutex 中毒后把 GPUI 前台回调拖到 abort。
- pane 构造发生在 GPUI non-unwinding 窗口回调内,spawn 失败必须优雅回退,不要 `expect()`。

### agent 进程与 shell-detected agent

- Windows 上 `claude` / `codex` 常是无扩展名 npm shim;直接 `CreateProcessW` 可能失败,需要经 shell / PATHEXT 路径启动。
- 持久 agent pane 托管在 `pwsh -NoExit` 时,agent 子命令退出不等于 PTY 子进程退出;需要明确的退出哨兵或 block 状态来清理 agent surface。
- shell 内手敲 agent 只能认当前运行命令首 token,不要靠进程树或“最近会话新鲜度”猜测。
- shell-detected agent 的 overlay 宿主必须保持 flex-column,否则正文区域会塌成“只有外壳没有正文”。
- Agent Host 身份走开放 `AgentId` + registry,UI 不重新引入闭合 `AgentKind` 特判。
- sidecar 遥测是 per-pane 资源,不能塞进共享 registry adapter;sidecar 是否联网也不等于 agent 本体是否远程运行。

### SSH / WSL / 远端命名空间

- WSL/SSH 的 `/home/...` 不能直接转成宿主 Windows `PathBuf`;必须经 `LaunchSpec::file_namespace` 显式区分 Host / WSL / SSH。
- SSH cwd 可能只通过裸 `OSC 633;P;Cwd` 更新模型级 cwd,不要只读当前 block / last finished block。
- SSH 覆盖卡存在时要前端阻断输入,后端在 `request_shell` 成功前 drain pending input,避免连接后 replay 到远端 shell。
- SSH 认证不能只尝试 `password`;很多服务器只接受 `keyboard-interactive`,需要按服务器开放方式尝试。
- known_hosts 写入失败不能静默继续;“记住主机”必须真正写入成功。
- SSH 连接器的数据源要分清:最近/收藏来自 Tn 自己的 `ssh_recents.json`,ssh-config Host 是只读 OpenSSH 配置入口。

### resize / scrollback / TUI 归属

- ConPTY 与 alacritty 尺寸不能长期不一致;行锁定会制造空白和光标错位。
- 长高 resize 要按 ConPTY repaint 语义处理 anchoring,避免把历史行拉进视口后又被 ConPTY 覆盖。
- 隐藏/恢复帧的小 bounds 不能被 clamp 成 `1x1` 真实终端尺寸;不足一个完整 cell / row 时跳过本帧 resize。
- scrollback 里不要画 live cursor;任何非 0 `display_offset` 输入前先回到底部。
- **【本节真因,先读这条,2026-06-14】** 系统 `conhost.exe`(Win10 build 19041)以 legacy 模式跑 ConPTY,会**内部吞掉**子进程的备用屏/鼠标 DECSET(1049/1003/1006/1007),只把重绘后的主屏回吐给宿主 → alacritty 永远看到 `alt_screen=false mouse_report=false history=0`(实测探针确认)。**下面「鼠标上报」「过滤 CSI 3J」「--no-alt-screen」「Tn 自管 transcript」几条都是在这个被污染的前提上打转,治标或永不触发。** Windows 终端能滚 codex,是因为它**自带新版 `OpenConsole.exe`(1.24)**,走 passthrough 把模式如实转发。修复:侧载 Microsoft 官方 `conpty.dll`+`OpenConsole.exe`(入库 `crates/tn-app/vendor/`,`build.rs` 拷到 `tn.exe` 旁;`portable-pty` 的 `load_conpty` 本就优先用 app 目录旁的 `conpty.dll`,免改 tn-pty)→ alacritty 拿到真实 `mouse_report=true`,既有鼠标上报路径自然把滚轮喂给 codex,原生回滚。**诊断教训:`alt_screen`/`mouse_report` 这类读数来自 alacritty,会被下层老 ConPTY 污染——别把宿主侧读数当子进程事实;要验证子进程真实模式,起子进程把 ConPTY 转发的原始字节 dump 出来,而不是信日志原地修 Tn 代码。**
- agent / alt-screen pane 的滚轮归程序自身,普通 shell 才滚 Tn scrollback。但「归程序自身」≠ 发方向键:接管鼠标的 TUI(codex = ratatui/crossterm,DECSET 1000/1002/1003+1006)要把滚轮编码成**鼠标上报**(按钮 64/65,SGR `ESC[<b;col;rowM` 或 legacy `ESC[M`+三个 +32 字节)发给 PTY,它才会滚自己的 transcript;只发方向键(`ESC[A/B`)它不认。alt-screen 没有回滚缓冲,放大 `scrollback_lines` 对这类 agent 完全无效——别拿主屏(Claude Code/Ink)的修法套全部 agent。`InputMode` 需暴露 `mouse_report`/`sgr_mouse` 让输入层判断。**前提是宿主能看见该模式——老 ConPTY 会吞掉它,务必先满足上一条(侧载新版 ConPTY)。**
- 主屏 agent(Codex/Claude/Ink 类)会用 `CSI 2 J` 重绘可见区,还可能跟 `CSI 3 J` 清除 saved lines;在 Tn 的 agent reader 中必须过滤 `CSI 3 J`,否则长输出会先进入 scrollback 又被清空,表现为无滚动条且鼠标滚轮无反应。普通 shell 不过滤,保留标准终端语义。
- 看到“最大化后历史变多,缩小后只能滚到最大化时可见的那几行”时,不要再从滚轮/滚动条入手:这说明 agent 只按当前 PTY 高度重绘自己的 transcript 视口,并没有把更早内容写入终端全局 scrollback。**【已取代,勿再走】** 曾据此让 agent 以 inline 模式启动(Codex `--no-alt-screen`、Claude `CLAUDE_CODE_DISABLE_ALTERNATE_SCREEN=1`),真机实测**打字/中文/IME 回归且 history 仍只十几行**,已撤回(`default_args/default_env` 这套 plumbing 保留备用)。真因不是 agent 渲染方式,而是老 ConPTY 吞掉了备用屏模式——见本节首条「真因」,正解是侧载新版 ConPTY 让 codex 用自己的滚轮回滚。
- 从 VS Code / Cursor 集成终端启动 Tn 时要清理会误导 agent 的 IDE 环境变量。
- **conhost 用 `ESC[40m` 表达「默认背景」**(legacy 控制台模型里默认底色 = color 0 黑):conhost 经 ConPTY 重发命令行时,把输入区每个 cell 的底**显式**设成 ANSI 黑。若主题 `ansi.black` ≠ `terminal.background`,这些 cell(尤其打一行字再删除后残留的空格)会渲染成一条「浅白色遮罩」带(2026-06-14 用户报、探针实证:唯一底色 SGR 就是 `ESC[40m`)。修法在 **`tn-core` `snapshot()`**:把**黑色背景**(`NamedColor::Black`/`Indexed(0)`)归一为默认背景(`color0 ≈ background` 通用约定),使其走「仅非默认底才实绘」被跳过、透出面板;黑色**前景**不动(深色文字仍可见)。**注意主题是首启写用户副本的(`theme.rs` 注释),改 toml 对存量用户无效,所以必须在代码层修**(同 scrollback 5k→50k 那次的坑)。
