# 基础性能与审查勘误

## 定位

本文承载 2026-06-02 至 2026-06-04 的基础性能优化、长期资源治理、构建配置补强和历史优化记录审查勘误。母索引只保留入口摘要。

## 背景

早期优化围绕终端大吞吐、GPUI/Rust 渲染热路径、后台轮询和资源释放展开。后续真机反馈表明部分记录存在“已落地但描述夸大”“后续实现覆盖旧实现”“优化未打中真因”的情况,因此 2026-06-04 对 1-41 项进行了集中审查和勘误。

## 具体内容

### 全局分配器与构建配置

- 2026-06-02 接入 `mimalloc` 作为 `tn-app` 全局分配器,缓解 Windows 默认分配器在大量小文本对象、跨线程 GPUI 渲染树构建时的碎片和锁竞争。
- Release profile 从 thin LTO 调整为 fat LTO,强化跨 crate 内联,服务于 VT 解析和渲染主循环等高频路径。
- 2026-06-04 勘误指出 release LTO 对日常 `cargo run` debug 构建无效。补强为 `[profile.dev] opt-level = 1` 加 `[profile.dev.package."*"] opt-level = 3`,覆盖 `alacritty_terminal`、`tn-core`、`gpui` 等终端热路径依赖。
- 图片、Office 解析依赖的 dev opt-level 覆写仍保留,但不再被视为唯一性能配方。

### 后台轮询与事件防抖

- Git watcher 早期从 450ms 提升到 1000ms,减少构建期间 `git diff` 刷新。
- 审查后改为 trailing-edge 防抖:持续文件事件会重置静默计时,源码单次保存约 300ms 响应,长构建期间只在停下后刷新一次。
- `is_noise_path` 过滤 `target`、`.git`、`node_modules` 等噪声目录,成为真正阻断构建刷屏的主力。
- AI usage poller 的“offset 增量读取”记录被判定为未落地。实际缓解来自将同步磁盘 I/O 隔离到 OS 线程,并由 dev profile 优化降低渲染热路径成本。

### 渲染缓存与小字符串优化

- 终端 `RenderCache` 通过 `generation` 前置判断跳过无变化帧的字符级 diff,降低待机 CPU。
- 审查确认无条件 bump generation 是保守正确取舍。使用 alacritty damage 精准化会漏掉 mode、光标显隐、滚动、cursor shape 等状态变化,容易出现陈旧显示。
- `SmolStr` 用于减少终端短 run 和 block metadata 的小字符串堆分配,但原记录中“Zero-Allocation”“终结 GC 分配”表述夸大。长同色 run 仍会堆分配,且存在 `String -> SmolStr` 拷贝。
- Quick Look 的 SmolStr 记录也被勘误:它主要用于高亮 span 缓存,并未替代文件列表、路径或 buffer 主体。

### 空闲内存回收

- 早期记录称 mimalloc 空闲页面回收发生在空闲循环,但实现曾是每 30s 无条件 `mi_collect(true)`。
- 审查后改为 PTY 活动计数驱动:reader 收到输出即递增 relaxed atomic;GC 线程每 5s 采样,只有连续约 10s 无 PTY 输出才强制 collect 一次。
- 繁忙终端不再被周期性强制 GC 打断,空闲窗口仍可把高水位内存页还给操作系统。

### 历史记录审查结论

- 1-41 项中多数实现真实落地,但多处描述被后续实现覆盖或措辞夸大。
- 失实或被覆盖的重点包括:Quick Look Fast-Path 已被取消令牌和全后台即时响应取代;Quick Look SmolStr 对象失实;usage offset 未落地;Quick Look 关闭并非从状态树强制析构 entity,而是 close 清空配合 `remove_asset`。
- 确证 bug 已在专题文档分别承接:Explorer git 前台阻塞、Quick Look 截断文件可编辑、edit cache 陈旧、scrollback swap 错误路径状态不一致、Excel two-pass 对齐未真正落地等。

## 验证 / 状态

- 相关阶段曾通过 `cargo test -p tn-ui --lib`、`cargo test --workspace --lib` 和 `cargo build --workspace`。
- dev profile 优化需要首次重编第三方依赖;后续本地 crate 增量编译仍可接受。
- 原有性能收益多为定性记录;部分百分比收益缺少独立 benchmark,只作为历史背景保留。
