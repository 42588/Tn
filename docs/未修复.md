## Quick Look 速览 / 编辑器

- **散文 / Markdown 长段落不能软换行**:固定行高 `uniform_list`(只 measure row 0)→ wrap 会变高破坏虚拟化,长段落只能横向滚。prose 体验欠佳。要 wrap 得放弃该场景虚拟化或改测每行高。
- **Diff tab 不支持拖选**:鼠标拖选只做了 File / 编辑态;Diff 行未接。
- **编辑态每键 `Rc::make_mut` 深拷整 buffer**:412 行亚毫秒、非冻结主因,但接近 `MAX_LINES`(4000)会有延迟感;真要治改 `Rc<RefCell<Vec<String>>>`。

### 技术评估方案 5: 自定义 GPUI Element 重写 Quick Look code renderer

目标不是把 Quick Look 做成完整 IDE,而是把当前 `uniform_list + div/span` 的代码区替换为一个原生绘制的 `QuickLookCodeElement`:文本、选区、搜索命中、caret、IME preedit、横向滚动条都在同一个 `Element::paint` 管线里完成。这样光标动画不再通过每帧重建可见行 UI 树实现,而是像终端 `TerminalElement` 一样只更新少量绘制状态。

**核心思路**

- **数据层保留现状**:`QuickLook` 仍持有 `buf: Vec<String>` / `cursor` / `sel_anchor` / undo/redo / find/replace / save;先不改编辑语义,降低风险。
- **渲染层替换**:新增 `quick_look::code_element` 模块,实现 GPUI `Element` 的 `layout/prepaint/paint`。`layout` 只报告容器尺寸;`prepaint` 计算可见行区间、行号槽、最长行宽、caret 目标 rect;`paint` 直接画背景、行号、文本 runs、选区、匹配、高亮、caret。
- **文本 shaping 缓存**:按 `(line_index, line_hash, theme_revision, font_key, mode)` 缓存 shaped runs。编辑态只失效改动行;预览态可长期复用。避免当前每行多个 `div`/span 造成的布局和 shaping 开销。
- **固定单元格模型继续保留**:ASCII 按 1 列、CJK/宽字符按 2 列;坐标仍以 `disp_width * char_w` 为准。这样点击定位、拖选、横滚、caret 宽度与现有修复保持一致。
- **动画 caret 内建**:Element 自己维护 `caret_px` / `target_px` / `anim_start` / `anim_kind`。小范围同行移动走 60-90ms glide + squash/pop;跨行、滚动、查找跳转、PageUp/Down、文件切换直接 snap。每帧只重绘 code canvas,不重建 Quick Look 外层 UI。
- **反相字符由 paint 层解决**:画 caret 时先画块底,再从当前行取 caret 下字符,用背景色重绘一次;行尾则只画空块。不会再出现叠层半途带目标字符导致“光标里字乱变”的问题。
- **滚动模型自管**:Element 持有 `scroll_y` / `hscroll_px`,滚轮、拖 scrollbar、caret-follow 都进入同一状态机。固定行高时可见行 = `floor(scroll_y / ROW_H)..`;未来软换行可扩展为 line height table。

**交互接入**

- 输入事件仍走 `QuickLook::on_key` 和现有 `op_*` 纯函数,先不把编辑器状态下沉到 Element。
- 鼠标事件由 Element 命中后回调 `QuickLook` 修改 cursor/selection/hscroll/scroll;或用小型 command enum 返回给 parent,避免在 paint 层改业务状态。
- IME candidate bounds 由 Element 根据 `cursor`、`scroll_y`、`hscroll_px` 暴露精确 rect;preedit 作为临时 run 在 paint 层覆盖。
- Diff tab 可复用同一 renderer 的只读模式:禁用编辑 caret,保留拖选、复制、diff 行着色。

**预期收益**

- 光标动画、选区、搜索命中、语法色全部变成绘制层状态,不再通过 UI 树拆分行内 span。
- 大文件滚动更稳定:可见行绘制 O(visible_rows),缓存命中时只做 paint。
- 软换行、Diff 拖选、多光标、列选等后续能力会有更清晰的落点。

**主要风险**

- GPUI 文本绘制 API/缓存细节需要真机 spike;若不能稳定复用 text shaping,收益会缩水。
- 自绘文本要自己处理字体 fallback、CJK 宽度、emoji/组合字符、tab 宽度、选择复制的视觉一致性。
- Element 级滚动/命中测试会替代 `uniform_list` 的成熟行为,需要补足边界测试。
- 这是中等偏大的渲染重构,不应和编辑语义重构(`Rc::make_mut` buffer 改造等)同批做。

**分阶段验证**

1. **Spike A: 只读 renderer**。在 File 预览态用自定义 Element 画文本 + 行号 + 纵横滚,不接编辑,与现有渲染做开关对比。
2. **Spike B: 命中测试 + 只读拖选**。实现鼠标定位、拖选、复制,同时补 ASCII/CJK/混排坐标单测。
3. **Spike C: 编辑态接入**。复用现有 `op_*`,接 cursor/selection/IME/preedit/caret-follow,先使用瞬时反相块。
4. **Spike D: caret 动画**。加入同线小步 glide + squash/pop,记录 `TN_PERF` 下连续输入、长中文行、4000 行文件的帧耗时。
5. **替换旧路径**。确认 File/编辑/Diff 的行为和性能都达标后,再移除旧 `edit_row_cached`/`file_row_cached` div/span 路径。

**通过标准**

- 连续输入 200 字符无明显卡顿,caret 不落后真实 cursor。
- 中文长行点击、拖选、横滚、IME candidate 位置与当前版本同等精确或更好。
- 4000 行文本滚动和编辑可用,无整面板每帧重建导致的 CPU 尖峰。
- File/编辑/Diff 三态键鼠交互无回归;`cargo test -p tn-ui` 保留坐标/编辑纯函数回归。

## 对面板进行解耦

目标是让每个终端窗格拥有自己的“工作区上下文”,而不是让左侧文件树、Agent 改动列表、用量统计都依赖一个全局状态。

- **每个 pane 绑定自己的文件上下文**:每个 Host / WSL / SSH pane 都应记录自己的文件命名空间、当前 cwd、对应的 Explorer root。切换焦点 pane 时,左侧文件树显示该 pane 对应的文件系统视图;切回其它 pane 时恢复它自己的目录状态、展开状态和选中文件。
- **Agent 信息跟 pane 绑定**:Claude / Codex 这类 agent pane 的“本次改动”、git diff、用量环、会话状态都必须归属到具体 pane,不能被其它 pane 的 cwd 或新启动的 agent 会话串台。
- **左上角「打开文件夹」按当前 pane 的命名空间执行**:
  - Host / Windows pane:继续使用系统原生文件夹选择器,选择 Windows 路径后更新当前 pane 的 Explorer root,并让对应本地 shell `cd` 到该目录。
  - WSL pane:用户应能像打开 Windows 文件夹一样浏览 Linux 目录,但实际显示的是 `\\wsl$\<distro>\...` 映射出的 Linux 文件树;选择目录后要转换回 Linux 路径,让对应 WSL bash pane `cd` 到该目录。
  - SSH pane:需要先实现 SFTP / 远端文件服务。完成后,「打开文件夹」应打开远端目录选择器或远端 Explorer,显示远端 Linux/macOS 目录;选择目录后让对应 SSH shell `cd` 到远端路径。
- **同步范围要收敛到目标 pane**:用户对某个 pane 打开的文件夹或执行的目录切换,默认只影响该 pane 及其绑定的文件树/Agent rail;不要无差别把 `cd` 发送给所有 pane,尤其不能发送给 Claude/Codex 交互中的 agent pane。
- **命名空间边界不能混淆**:Windows `PathBuf`、WSL Linux 路径、SSH 远端路径必须显式转换。没有远端 FS 后端时,SSH cwd 只能作为终端状态展示,不能驱动本机 Explorer / Quick Look / git watcher。
