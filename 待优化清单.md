## 一、总览

Tn 是一个 Rust + GPUI 的 GPU 加速 Windows 终端，设计目标清晰，架构分层合理 (headless crate 严格隔离 GPUI)，代码整体质量高。93 个单测全部通过。以下按严重程度 (高/中/低) 列出待优化项。

## 二、性能

### 2.1 【高】终端网格每帧全量重建

[terminal_view.rs](crates/tn-ui/src/terminal_view.rs:1) 的 `render()` 方法中，每一帧都调用 `snapshot().row_runs()` 遍历整个网格 (rows x cols)，并为每个 `CellRun` 生成一个 GPUI `div`。对于 34x110 的默认尺寸，每帧至少创建上千个 div。GPUI 内部 diff 可以处理，但分配开销仍然可观。

**建议**: 引入脏区域追踪 — 只重建自上次渲染以来发生变化的行。alacritty 的 `damage` 机制已经提供了变更行号，但当前 `snapshot()` 丢弃了这些信息。

### 2.2 【高】workspace render 无缓存

[workspace.rs](crates/tn-ui/src/workspace.rs:1) 的 `render()` 每帧重新计算 tab 信息 (`tab_info`)、重建 tab 栏和 status bar。这些数据在大部分帧中不变。

**建议**: 将 tab 信息缓存到 `Workspace` 状态中，仅在 tab 切换/pane 变化时更新。

### 2.3 【中】Reader 线程缓冲区固定 8KB

[terminal_view.rs](crates/tn-ui/src/terminal_view.rs:1) `spawn_reader` 使用 `[0u8; 8192]` 固定缓冲区。高吞吐场景 (cat 大文件、编译输出) 时读调用次数多，锁竞争高频。

**建议**: 增大到 64KB 或使用 BufReader 动态缓冲。`Mutex<Box<dyn Read+Send>>` 可以用 `BufReader::with_capacity(65536, reader)` 包裹。

### 2.4 【中】cursor_abs_line 每批调用

[terminal_view.rs](crates/tn-ui/src/terminal_view.rs:1) reader 线程在每次 `advance` 后调用 `t.cursor_abs_line()`，该方法内部调用 `self.term.grid()` 触发 alacritty 的 borrow。对于批处理场景这是不必要的锁开销。

**建议**: 如果当前批 `BlockEvent` 列表为空 (绝大多数情况)，跳过 `cursor_abs_line()`。

### 2.5 【低】explorer 每帧重建 row 数据

[explorer.rs](crates/tn-ui/src/explorer.rs:1) `render()` 中为每个 row 克隆 `path`/`name` 并重建 `Row` 结构体。rows 缓存了，但 `render_row()` 的参数仍然是 clone 出来的。

**建议**: 直接引用 `self.rows[i]` 传入 `render_row`，避免 clone。

## 三、用户体验

### 3.1 【高】缺少光标闪烁

CLAUDE.md 中标注为 "待后置"，但这是用户最直观的终端体验问题。光标恒定常亮会让人无法区分"终端挂了"和"正在等待输入"。

**建议**: 实现 500ms 间歇闪烁，关联 GPUI 的 frame clock。

### 3.2 【高】无滚动条

用户无法直观感知 scrollback 中的位置，也无法拖拽跳转。纯键盘滚动 (ScrollWheel) 对触控板和长历史用户不够友好。

**建议**: 在 term_area 右侧渲染一个 ~6px 宽的滚动条指示器，显示当前位置 / 总历史比例。

### 3.3 【高】Quick Terminal 动画使用了线性而非 ease-out

[quick_terminal.rs](crates/tn-ui/src/quick_terminal.rs:1) `slide()` 使用 `progress` 和 `1.0 - progress` 做线性插值，但 `tn-config::quick_terminal` 里已经实现了 `ease_out_cubic` 函数却未使用。

**建议**: 将 `tn_config::quick_terminal::ease_out_cubic` 暴露并在 slide 循环中调用，让下滑动画有自然的减速感。

### 3.4 【中】终端无搜索功能

用户无法在 scrollback 中搜索文本。这是成熟终端的基本功能。

**建议**: `tn-core::Terminal` 添加 `search(query: &str) -> Vec<(usize, usize)>` 方法，UI 层实现 Ctrl+Shift+F 搜索栏。

### 3.5 【中】选择仅支持 Simple 模式

[lib.rs](crates/tn-core/src/lib.rs:1) 选区内只使用 `SelectionType::Simple`。不支持双击选词、三击选行。

**建议**: 在 mouse_down handler 中记录点击次数和时间，对双击/三击切换到 `SelectionType::Semantic` / `SelectionType::Lines`。

### 3.6 【中】无 URL 检测/悬停高亮

终端中的 URL 无法点击打开，也不会高亮。这是 Warp/Windows Terminal 的标配。

**建议**: 在 snapshot 构建时扫描可见行中的 URL 模式，对 URL cell 标记 `is_url` 标志，渲染时加下划线，Ctrl+Click 用 `ShellExecuteW` 打开。

### 3.7 【中】Explorer 无自动刷新

[explorer.rs](crates/tn-ui/src/explorer.rs:1) 在 `new()` 时 `rebuild()` 一次后不再更新。用户切换 git 分支、新建文件后 explorer 不反映变化。

**建议**: 集成 `notify` crate 监听根目录文件变更，自动 `rebuild()`。

### 3.8 【中】终端响铃被忽略

[terminal_view.rs](crates/tn-ui/src/terminal_view.rs:1) `drain_events` 中对 `TermEvent::Bell` 作 `_ => {}` 忽略。

**建议**: 至少实现视觉响铃 (背景短暂闪烁)，可选音频 bell 由配置控制。

### 3.9 【低】终端 resize 无过渡动画

[terminal_view.rs](crates/tn-ui/src/terminal_view.rs:1) 网格尺寸在 `render()` 中计算后立即 `self.size = new_size` 并 `resize()`，分屏拖拽时网格跳跃而不平滑。

**建议**: 对 `cell_width` 和 `line_height` 做插值过渡，但注意这需要字体渲染支持非整数像素。

### 3.10 【低】无 pane 切换动画

[workspace.rs](crates/tn-ui/src/workspace.rs:1) `activate_tab` 直接替换 active 索引，无过渡效果。

**建议**: 添加 focus border 的颜色渐变 (从当前 accent 平滑过渡到新 pane 的 accent)。

## 四、界面设计

### 4.1 【高】Calm Glass 常量在 6 个文件中重复定义

`RIM`/`SHEEN`/`INSET`/`HOVER`/`R_WINDOW`/`R_PANEL`/`R_CARD`/`UI_SANS` 以及 `col()`/`cola()`/`soft_shadow()`/`shadowed()` 工具函数在以下文件中各自定义:

- `workspace.rs`
- `terminal_view.rs`
- `quick_terminal.rs`
- `explorer.rs`
- `viewer.rs`
- `block_view.rs`

任何视觉微调需要改 6 处，极易遗漏。

**建议**: 在 `tn-ui/src/theme.rs` 或 `tn-ui/src/lib.rs` 中集中定义 Calm Glass tokens 和工具函数，各模块通过 `use crate::...` 引用。

### 4.2 【中】指令面板与 Quick Terminal 启动器代码几乎重复

[workspace.rs](crates/tn-ui/src/workspace.rs:1) 的 `render_palette()` 和 [quick_terminal.rs](crates/tn-ui/src/quick_terminal.rs:1) 的 `render_picker()` 有大量相同的结构 (搜索栏、profile 行、高亮逻辑、键盘导航)。

**建议**: 抽取 `picker.rs` 共享组件，接受 `on_select` 回调和 profile 列表。

### 4.3 【中】Viewer 的 Diff/File tab 点击区域是隐形覆盖层

[viewer.rs](crates/tn-ui/src/viewer.rs:1) 的 `render()` 中，clickable tab chips 不是直接绑在 header 的 chip 上，而是在 header 上方覆盖了不可见的 `div` (34x20 px 透明区域) 作为点击目标。这是 workaround，脆弱且难维护。

**建议**: 在 `render_header()` 中直接给 chip `div` 绑 `on_mouse_down`，或重构 header 为 `render()` 方法而非独立函数。

### 4.4 【低】Explorer 中选中高亮色值硬编码

[explorer.rs](crates/tn-ui/src/explorer.rs:1) 的 hover 和选中背景使用硬编码 `rgba(0xffffff14)` / `rgba(0xffffff0a)`，而非从主题读取。

**建议**: 使用 `cola(ui.selected_bg, 0.14)` 等主题色，保持全局一致性。

### 4.5 【低】无自适应布局

当前所有面板宽度 (Explorer 214px, Viewer 420px, Palette 540px) 都是硬编码 `px`。窄窗口时这些面板可能无法完整显示。

**建议**: 关键面板宽度使用 `clamp` 或百分比，或在窄窗口时自动折叠侧栏。

## 五、动画与交互

### 5.1 【中】Quick Terminal 动画帧率固定 62.5fps

[quick_terminal.rs](crates/tn-ui/src/quick_terminal.rs:1) 的 slide 循环使用 `timer(Duration::from_millis(16)).await`，在高刷新率 (120/144Hz) 显示器上浪费了帧预算。

**建议**: 改为基于 `request_animation_frame` 或 GPUI 的 frame callback，或至少缩短到 8ms。

### 5.2 【中】动画被打断时无正确处理

[quick_terminal.rs](crates/tn-ui/src/quick_terminal.rs:1) 通过 `anim_token` 检测是否有新动画启动来取消旧动画，但取消时目标窗口已处于中间状态。

**建议**: 被打断时立即 snap 到最终位置，或从当前位置开始新动画 (需要记录当前进度)。

### 5.3 【低】无 block bar 出现/消失动画

[block_view.rs](crates/tn-ui/src/block_view.rs:1) 的 block 栏在命令开始/结束时直接出现和消失。

**建议**: 添加 150ms 滑入/滑出过渡。

### 5.4 【低】无 Alt-Screen 切换过渡

[terminal_view.rs](crates/tn-ui/src/terminal_view.rs:1) alt-screen 模式切换 (进入 vim/退出 vim) 时网格直接替换，视觉上是闪烁。

**建议**: 添加 ~100ms 的 crossfade，让全屏应用的进入/退出更平滑。

## 六、代码架构

### 6.1 【高】workspace.rs 过于庞大 (1575 行)

[workspace.rs](crates/tn-ui/src/workspace.rs:1) 包含了: tab 管理、pane 树、指令面板、状态栏、编译演示、整个 layout 渲染。违反单一职责。

**建议拆分**:
- `tab.rs`: `Tab` + `Node` + pane tree 操作 (split/prune/resize)
- `palette.rs`: 指令面板 UI + 键盘导航
- `status_bar.rs`: 底部状态栏
- `workspace.rs`: 只剩顶层编排 (max ~400 行)

### 6.2 【高】terminal_view.rs 过于庞大 (1186 行)

[terminal_view.rs](crates/tn-ui/src/terminal_view.rs:1) 承担了: PTY spawn、reader 线程管理、repaint loop、exit watcher、usage poller、渲染、输入编码桥接、block bar 交互、鼠标选择、滚动。

**建议拆分**:
- `pty_bridge.rs`: reader 线程 + repaint channel + exit watcher
- `agent_header.rs`: agent 头部 UI (render_agent_header / usage_ring)
- `terminal_view.rs`: 只保留渲染核心

### 6.3 【中】LaunchSpec::from_profile_inner 逻辑分支复杂

[terminal_view.rs](crates/tn-ui/src/terminal_view.rs:1) 的 `from_profile_inner` 同时处理 WSL / SSH / 原生 pwsh / pwsh 托管四种路径，函数体 ~100 行。

**建议**: 每种 `ProfileKind` 拆分到独立函数: `launch_wsl()`, `launch_ssh()`, `launch_pwsh()`, `launch_hosted()`。

### 6.4 【中】无统一的 event bus

`UsageUpdated` 和 `ProcessExited` 是通过 GPUI 的 `cx.emit()` + `cx.subscribe()` 传递，但跨组件通信 (如 Explorer -> Viewer 的 `OpenFile`) 也是同样模式。当事件流增多时，workspace 的 `cx.subscribe` 调用会膨胀。

**建议**: 不需要现在就改，但关注 GPUI 是否有 `cx.observe_global` 机制可以用于全局事件总线。

### 6.5 【低】SSH backend 同步启动

[ssh.rs](crates/tn-pty/src/ssh.rs:1) 的 `SshBackend::spawn` 在构造时同步建立 russh 连接，阻塞 GPUI 的主线程回调。

**建议**: 改为异步连接 — spawn 返回一个 pending backend，在后台任务中建立连接，连接就绪后通知 UI。

## 七、测试覆盖

### 7.1 【高】零集成测试

当前只有 93 个单元测试，全部运行在 headless crate 中。`tn-app` 和 `tn-cli` 的测试数是 0。没有端到端测试验证:

- 完整 ConPTY -> Terminal -> 渲染 pipeline
- 键盘输入 -> 编码 -> PTY -> 输出 -> 解析链路
- 分屏 / tab 切换后 pane 聚焦正确性
- 配置热重载的健壮性

**建议**: 至少添加 `TN_AUTOQUIT` 模式的 CI 测试，验证: 启动 -> 注入命令 -> 等待输出 -> 验证网格文本 -> 退出码 0。

### 7.2 【中】无 snapshot 测试

[lib.rs](crates/tn-core/src/lib.rs:1) 的 VTE 解析已有单测，但没有 golden-file 风格的 snapshot 测试来防止回归。VT 解析器非常容易在升级 alacritty_terminal 版本时引入行为变化。

**建议**: 对已知的 ANSI 序列输入集，保存 `snapshot().to_text()` 的期望输出到文件，对比自动化。

### 7.3 【中】无性能回归测试

没有任何 benchmark 或性能测试。无法量化 "某次优化提升/恶化了多少帧率"。

**建议**: 使用 `cargo bench` 或 criterion 对以下路径做 benchmark:
- `Terminal::advance()` 吞吐量
- `TerminalSnapshot::row_runs()` 生成速度
- `encode_key()` 延迟

### 7.4 【低】tn-ui 测试仅覆盖输入编码和分屏 resize

[workspace.rs](crates/tn-ui/src/workspace.rs:1) 的 3 个测试只测 `Node::resize`，[input.rs](crates/tn-ui/src/input.rs:1) 的 10 个测试只测编码。Explorer, Viewer, BlockBar, Palette, QuickTerminal 无任何测试。

**建议**: 至少对 explorer 的 reconstruction、block model 的序列化和 bar base 的构建加测试。

### 7.5 【低】输入编码无模糊测试

[input.rs](crates/tn-ui/src/input.rs:1) 的 `encode_key` 返回 `Option<Vec<u8>>`，有大量分支。没有 100% 覆盖。

**建议**: 添加 property-based test: "任意 Keystroke 输入，encode_key 要么返回有效字节序列，要么返回 None，不应 panic"。

## 八、健壮性与错误处理

### 8.1 【中】reader 线程 panic 会导致静默停止

[terminal_view.rs](crates/tn-ui/src/terminal_view.rs:1) `spawn_reader` 以 `thread::spawn` 启动，无 panic catch。如果 alacritty 内部 panic (罕见但可能)，整个 reader 静默死亡，终端不再更新。

**建议**: 使用 `std::panic::catch_unwind` 包裹 reader 循环，panic 时 `tracing::error!` 记录并 break (让 exit watcher 检测)。

### 8.2 【中】git_branch 和 compute_git_status 无错误日志

[workspace.rs](crates/tn-ui/src/workspace.rs:1) 和 [explorer.rs](crates/tn-ui/src/explorer.rs:1) 中调用 `git` 命令失败时静默返回 `None`/空 HashMap。用户无法知道是 "不在 git repo" 还是 "git 命令未安装/权限问题"。

**建议**: 首次调用失败时 `tracing::warn!` 记录原因，后续静默。

### 8.3 【低】Codex session 解析无版本兼容性保障

[codex.rs](crates/tn-ai/src/codex.rs:1) 的 JSONL 解析大量使用 `v.get("field_name")` 和 `as_u64()/as_str()` 的链式调用。Codex 更新其日志格式字段名或类型时，现有解析会静默返回 0 或 None。

**建议**: 添加一个单元测试，使用真实的 Codex rollout 快照文件，确保不会静默退化。

## 九、文档

### 9.1 【中】CLAUDE.md 编码损坏

[CLAUDE.md](CLAUDE.md:1) 文件内容以正确的 UTF-8 存储，但在本次会话中读取时出现大量乱码 (如 "鈥?" 替代 "—")。这表明文件本身可能正常，但编码检测或读取工具有问题。

**建议**: 确认 CLAUDE.md 用的是 UTF-8 without BOM，并在仓库中加 `.editorconfig` 声明 `charset = utf-8`。

### 9.2 【低】API 文档缺失

所有 `pub` 函数都有 `//!` 模块文档，但一些重要的 pub 方法缺少 `///` 注释 (如 `Terminal::snapshot()`、`LaunchSpec::from_profile()`)。

**建议**: 补全关键 API 的 doc comment，启用 `#![warn(missing_docs)]`。
