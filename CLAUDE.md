# CLAUDE.md — Tn 终端(Claude Code 项目指南)

Tn 是一个 **Windows 优先、Rust 编写、GPU 加速、为 vibe coding 打造的终端**:把 AI 编码 CLI
(Claude Code、Codex)当一等公民托管,提供出色 UX、灵活平铺、WSL + SSH。技术栈:**GPUI**
(Zed 的框架,Windows 上走 DirectX 11 + DirectWrite)· **alacritty_terminal**(VT 引擎)·
**portable-pty**(ConPTY/WSL)+ **russh**(SSH)。许可证:GPL-3.0-or-later。

## 协作约定(最优先,先读)

- **每次回复结构**:先**规划** → 再**划分任务** → 再**执行** → 再**测试** → 最后**输出**。
- **默认中文**回答。
- **文档同步(单一真源,不重复)**:每个有效优化/设计**完成即更新**——`docs/架构蓝图.md`(路线图 + 架构层"做了什么/为什么")· `docs/产品设计.md`(用户可见的 UX/交互/视觉决策)· 本 `CLAUDE.md`(操作指南 + 踩过的坑)。**三处各司其职、互相引用而非复制**;逐项明细只进 ` [CHANGELOG.md](CHANGELOG.md)。
- **上下文压缩前记录**:每次预计会发生上下文压缩前,必须先把上一轮已执行操作追加到 `AGENTS.md` 末尾的「上下文压缩记录」,写清目标、关键改动、验证结果和未完成事项。
- **节奏**:完成一轮任务后**自动提交**(提交规范见文末「约定」)。

## 图形界面 / 样式开发(**必走此路**)

任何 UI / 颜值改动都沿这条链走,别绕:

1. **设计真源 = [`design/mockup.html`](design/mockup.html)**(浏览器打开即标准)。要改样子,先改/看 mockup。**逐面板原型** = [`design/panels/`](design/panels/)(共享 [`design/calm-glass.css`](design/calm-glass.css),入口 `index.html`);**面板 ↔ 真实实现 清单** = [`docs/产品设计.md`](docs/产品设计.md) §6.4。
2. **查确切值 → [`docs/样式还原手册.md`](docs/样式还原手册.md)**:§1–§15 是 CSS→gpui 译法(**HOW**),**§16(自动生成)是每个组件的权威数值(**WHAT**)——照抄不估**(§16.2 组件值从 [`design/calm-glass.css`](design/calm-glass.css) 抽取 = 组件全集含 quicklook/活动栏/app 菜单/welcome,令牌从主题)。改了 calm-glass/mockup/主题后跑 `TN_GEN_SPEC=1 cargo test -p tn-ui spec_gen` 刷新 §16。
3. **写 gpui 必用 token、禁止硬编码主题色**:颜色一律 `col(ui.xxx)` / `cola(色, a)`(半透明 alpha 写小数,**别手算 hex**);圆角/间距用 `style.rs` 令牌或精确 `px()`;字重 `FontWeight(N.)` 精确(gpui 接受任意 f32)。
4. **四道机器守卫**(`cargo test -p tn-ui`,headless 即可跑):
   - `style::token_drift` —— mockup `:root` 令牌 ↔ 主题 / `style.rs` 常量一致(漂了即红、点名);
   - `style::token_drift::mockup_and_calm_glass_roots_mirror` —— mockup `:root` ↔ calm-glass `:root` 镜像一致(两份手动同步的副本,改色/令牌须同步两边);
   - `style::no_hardcoded_theme_colors` —— UI 代码出现"RGB 命中主题色的裸 hex"即红、指出 file:line + 该用哪个 `col/cola`;
   - `style::spec_gen` —— 生成 / 校验 §16。
   - 合理例外(白叠加非整档 / fg-dim·faint 无 token / g1 渐变基色 / 透明 / scrim):裸 hex 但**带 `round(α×255)` 算式注释**。
5. **真机肉眼验**:`cargo run -p tn-app` 跑真实 app,与浏览器打开的 [`design/mockup.html`](design/mockup.html) / [`design/panels/`](design/panels/) 原型并排比(headless 环境用 `TN_AUTOQUIT` 只验网格收敛;颜值/动画须真机)。

> **现状**:全 UI 已逐组件对齐 mockup、主题色硬编码清零、四道守卫把关;`design/panels/` 新原型的主界面已 1:1 端口(agent 活动栏「本次改动」接真实 `git diff HEAD`、欢迎 launchpad + 应用内 Agent 编辑器、Quick Look 速览浮层 + 自绘编辑器、app 菜单 popup、分屏启动器、布局管理器)。「做了什么 / 为什么」见 [架构蓝图 §8](docs/架构蓝图.md) + [产品设计 §6.4](docs/产品设计.md),逐项明细见 [CHANGELOG.md](CHANGELOG.md);余欢迎页最近目录待接。**窗口保持 Opaque**(试过默认 acrylic 透背被 owner 否,见坑)。
> **面板解耦(per-pane 工作区)**:每个分屏 pane 自带文件树状态(`ExplorerSnapshot`;Workspace 按 `PaneId` 存 `explorer_states` / `explorer_pane`)。render 焦点跟随**分两路**——**换 pane** → `switch_pane`(存/取快照),**同 pane `cd`** → `follow_root`(保子目录展开态),**别混成一个**(那样跨 pane 切换会揉掉展开态)。「打开文件夹」`cd_pane_to_root` 只动焦点 pane;agent 用量 /「本次改动」/ git watcher 早已 per-pane(在 `TerminalView`)。**欢迎页焦点「pane」= `PaneId::MAX` 占位 dummy**(无 `LaunchSpec`,防与真实 pane 撞 + 关页误删 pane 0 致 abort;故「打开文件夹」对无 spec 用原生选择器,见坑)。

## 构建 / 运行 / 测试

`cargo` 在 `%USERPROFILE%\.cargo\bin\cargo.exe`(不在 bash PATH 上——PowerShell 里用全路径,或新开 shell)。

```powershell
cargo build --workspace
cargo test  --workspace --lib                  # 全 crate 库单测(headless)+ `--test conpty_pipeline` 1 集成
#   注:`--lib` 只跑库单测;去掉 `--lib` 还会跑集成测试(`conpty_pipeline`,需起 ConPTY)。设计复刻 example 已移除(原型改对照 design/panels)
$env:TN_PERF="1"; cargo run -p tn-app          # 开渲染性能日志(tn::perf,~1/s:渲染率/缓存命中率/重建耗时)
cargo bench -p tn-core --bench engine          # criterion 热路径基准:advance(VT 解析)+ snapshot/row_runs(每帧渲染数据)
cargo run   -p tn-cli                          # ConPTY 烟雾测试:起 shell、把网格渲染到 stdout、PASS/FAIL
cargo run   -p tn-cli -- wsl.exe -d Ubuntu -- echo HELLO_TN_MARKER   # 验证 WSL 经 ConPTY 跑通
cargo run   -p tn-app                          # 开终端窗口
$env:TN_AUTOQUIT="1"; cargo run -p tn-app      # headless 自测:首个 pane 跑命令、dump 网格、退出(exit 0)
$env:TN_DEMO="1";     cargo run -p tn-app      # 演示:窗口里自动步进滚动/选区/分屏/改尺寸,然后退出
```

> **dev 构建的运行性能(开发期跑高频重绘 TUI 必读)**:根 `Cargo.toml` 用 `[profile.dev] opt-level=1` + `[profile.dev.package."*"] opt-level=3`(Bevy「optimize deps, debug your code」配方)。默认 dev 全 opt0,而 alacritty VT `advance` / tn-core `snapshot`·`row_runs` / gpui 布局·paint 是 Claude Code(Ink,每秒数十次整屏重绘)的热路径 → opt0 下慢几十倍,直接表现为输入卡顿 + 面板拉起慢,且把零分配/channel/增量解析等逻辑优化的收益**完全淹没**。代价:首次 `cargo build` 重编依赖一次,之后改 tn-ui/tn-app 增量照常快。**(2026-06 复盘「做了很多优化依旧卡」的决定性真因。)**

## 踩过的坑(hard-won)

**ConPTY / 进程**

- **ConPTY DSR**:必须把 alacritty `Event::PtyWrite` 回复写回 PTY writer(回应启动时的 `ESC[6n`),否则子进程卡住(只读 4 字节)。ConPTY 无可靠 EOF——用 `try_wait` 轮询而非 `read()==0`。整个会话保活 portable-pty 的 `SlavePty`。
- **Windows 上 `claude`/`codex` 是无扩展名 npm shim**:`CreateProcessW` 直接拉起会 os error 193。**用 pwsh 托管**——`powershell -NoExit -Command "& 'codex'"`(走 PATHEXT 解析 `.cmd`,agent 退出后回 prompt)。
- **pane 构造在 GPUI 窗口回调 = non-unwinding**:`spawn(...).expect()` panic 会让整进程 **abort**(0xc0000409),不被 panic hook 接住。所以 spawn 失败**必须优雅回退**(回退 pwsh),绝不能 panic。
- **首个 pane 在 tab push *之前* spawn → `spawn_pane_with` 不能硬索引 `self.tabs[active]`**:构造器在 `TN_AUTOQUIT`/`TN_DEMO` 下先 `spawn_pane`(拿 id)再 `tabs.push(Tab::panes(id))`(顺序无法反转——`Tab::panes` 要 id)。而 `spawn_pane_with` 的 cwd 继承闭包(「用活动 pane 的 cwd」特性后加)`self.tabs[self.active].focused` 在此时 tabs 为空 → **越界 panic**(`len 0 index 0`,exit 101),只在这条首-pane 路径触发(正常路径先 push welcome tab 故长期没暴露,直到跑 `TN_AUTOQUIT` 才现形)。**修**:闭包改 `self.tabs.get(self.active).and_then(|t| self.panes.get(&t.focused))…`,tabs 空时回退 explorer root。**通则:任何在「首个 pane / tab 尚未建」窗口运行的代码都要 `tabs.get` 不硬索引。**
- **portable-pty `LocalPty` Drop 不杀子进程**:给 `LocalPty` 加 `Drop → child.clone_killer().kill()`;关标签时 drop 视图链(`panes.remove` → drop `TerminalView` → drop `Arc<...>` → kill)。
- **主窗口 agent 窗格托管在 `pwsh -NoExit` 里,退出 agent 后没有 `ProcessExited`**:`-NoExit` 让 pwsh 在 claude/codex 退出后存活(窗格变回 shell),但 PTY 子进程(pwsh)没死 → `spawn_exit_watcher` 不触发,`self.agent` 永不清,头部+陈旧用量一直挂着。**解法**:持久 agent 的启动命令在 `& 'claude'` 后追加 `; $Host.UI.RawUI.WindowTitle = 'TN::agent-exited'`(`AGENT_EXIT_SENTINEL`)——agent 退出后 `-NoExit` 的 pwsh 执行它、发出 OSC 标题哨兵;reader 在字节流里**有序**看到它(无竞争,即便随后 prompt 又改标题)→ 置原子位 → 前台 `clear_agent_if_exited` 清 `agent`/`usage`、标签回落 shell 名。ephemeral(快捷终端)**不**追加(它走 `ProcessExited` 回启动器)。别用进程树轮询找 grandchild(脆弱);哨兵标题确定且复用既有 OSC 通路。
- **「敲命令起 agent」自动切态 = 认 shell 集成的运行命令,别认进程树/会话新鲜度**:在普通 shell 里**敲** `claude`/`codex`(非经启动器)应切到 agent 头/活动栏。`agent` 默认**只认 launch intent**,所以要补检测——但**别**走进程树轮询(脆弱)或"会话新鲜度反推"(会把同目录的 dev agent 误标,见下条 codex 坑)。**解法**(`sync_shell_agent`,前台 repaint loop 里跑):读 `BlockModel::current()` 的**当前运行命令**(shell 集成 OSC 633;E 的命令行,本就在解析),取**首个 token**(程序名)过 `registry.match_command`(Agent Host 后:经注册表的命令别名匹配,**非**旧的 `agent_kind_for_command` 闭合枚举)——**只判程序名**故 `cd claude-proj` / `cat codex.md` 不误触。命中(得 `AgentId`)→ 置 `agent` + `agent_from_shell=true`、`resolve_agent_view` 解析 accent/label/caps、**有 adapter 才**起用量轮询 + 活动栏监听 + 发 `UsageUpdated` 重标签;命令**块结束**(`current()` 不再 running)即 `clear_agent` 还原。两类 agent 分开清:**launcher agent** 靠退出哨兵(上条),**shell agent** 靠命令块结束(`agent_from_shell` 区分),互不干扰(launcher 的命令首 token 是 `&`、不命中)。诚实——用户真敲了这命令。
- **Agent Host 平台化(2026-06):agent 身份走 `AgentId` + `AgentRegistry`(gpui Global),UI 零 `AgentKind` 特判**。要点与坑:① **注册表是 gpui Global**(`agent_host::AgentHost`,`run()` 装入),读经 `agent_registry(cx)` 返回**克隆**(Arc-backed Vec,廉价;避免持 `&App` 借用跨视图改);未装时返回**空注册表**→ agent 特性优雅降级(headless 测试不 panic)。② **空注册表 = 纯 shell 宿主**(P6 出厂态):无内置 agent;`tn_ai::builtin_registry()`(Claude+Codex)保留但**默认不注册**,用户经 config `[[agents]]`(generic 无遥测)或重新注册 adapter(telemetry)加回。③ **热路径不动**:用量轮询和外部实时事件都经 `reduce_agent_event` 收敛成单一 UI 输入流;内置日志 adapter 复用既有后台轮询线程,只有 `has_realtime_events()` 的 adapter 才启动 120ms 轻量 drain poller。④ **外部事件必须来自明确 adapter 信号,不从 PTY 猜状态**:`ExternalEventAdapter` 吃 JSONL、`ExternalProcessAdapter` 拉起 stdio 子进程读 stdout JSONL;`StatusChanged`/`ModelChanged`/`TranscriptAppended`/`PermissionRequested`/`ErrorReported` 才会在 header 显示状态/权限/错误/摘要 chip,普通 PTY agent 仍不伪造 Thinking。⑤ **per-pane 解析缓存**:`agent_accent`/`agent_label`/`agent_short`/`agent_manages_cursor`/`agent_caps` 只在 agent **变更**时经 `resolve_agent_view`(descriptor + theme + adapter)重算,render 路径只读缓存、不碰注册表;`manages_own_cursor` 取代旧的 Claude 专属 `force_hide_cursor`。⑥ **capability 即插槽**:header 用量环 gate `caps.usage`、活动栏 gate `caps.git_diff`——无能力的 agent 隐藏对应区而非空着。⑦ **runtime 安全契约**:`[[agents]].runtime_support` 可声明 PTY / structured / http / websocket / remote-daemon;网络 runtime 默认拒绝,`allow_network=true` 也只是进入“需用户确认”的 `AgentNetworkPolicy::Ask`,不是静默联网。⑧ **守卫**:`agent_host::guard::ui_has_no_closed_agent_enum` 扫 `tn-ui/src` 禁止 `Agent`+`Kind`(needle 运行时拼接避免自匹配),锁死「UI 不得重新引入闭合枚举」。⑨ **层次铁律**:平台 `tn-agent` 对具体 agent 零知识;内置 Claude/Codex 只住 `tn-ai`(允许认它俩);`AgentRuntimeKind`(`LaunchSpec::runtime()` 派生)≠ `FileNamespace`,SSH runtime 的文件 I/O 只能经 SFTP-backed `ExplorerRoot::Ssh` / `RemoteFileService`,不得落回本机 `PathBuf` / git watcher。
- **应用内 Agent 编辑器(2026-06):加/改/删 agent 走「写 config 文件 → 重读 → 重建注册表 global + 重建 welcome 实体」,别手动改 `Arc<Loaded>`**。要点与坑:① **持久化复用 profiles 的"片段追加/块级删除"**(`tn_config::append_agent`/`remove_agent`,泛化 `block_ranges` 吃 `[[profiles]]`/`[[agents]]` 两种头)→ 保用户注释;**编辑 = remove(old_id) + append**(id 不变也这么做,保单条)。② **`reload_agents` 是唯一生效路径**:`tn_config::load()` 重读刚写的盘 → 重建 `AgentRegistry` 用 `cx.set_global(AgentHost(..))` **重装** global(不是手改 `self.config` 的 Arc,那只读)→ `discover_profiles` 刷 `launch_profiles` → **`cx.new` 重建 welcome 实体**。③ **重建 welcome 必须重新订阅**(抽 `Workspace::subscribe_welcome` 关联函数,`new()` 与 `reload_agents` 共用)——旧 welcome 实体丢弃后其 detached 订阅自然失效(emit 源没了),新实体若不订阅则点磁贴/加 agent 全哑。④ **名称字段中文 IME 复用 SSH 重命名的 `EntityInputHandler`**:6 个方法按 `agent_form_field==Name`(`agent_name_is_ime_target`)多路复用到 `agent_form_marked` vs `ssh_rename_marked`;命令字段 ASCII(render 里 `disable_ime` 把 `agent_form_open && field==Command` 也算进 IME 关条件);IME 开关跟踪字段从 `ssh_ime_disabled` 泛化重命名为 `ime_disabled`。⑤ **颜色预设放 `tn_config::ACCENT_SWATCHES`**(不放 tn-ui),tn-ui 经 `col()`/`cola()` 渲染 → 绕开 `no_hardcoded_theme_colors` 守卫(它只扫 tn-ui 的 `rgb(0x..)`/`rgba(0x..)` 字面量)。⑥ **进阶项也在 GUI**(2026-06):编辑器收 名称/命令/颜色/Ink 自绘光标 + **「高级 · 遥测 sidecar」**(`AgentField::Sidecar` 字段 + 「联网 sidecar」开关)。留空 sidecar = generic(无遥测、`capabilities=[]`);**设了 sidecar** → `save_agent_form` 自动 `capabilities=["usage"]`(解锁用量环;status/permission/transcript chip 不 gate,由 sidecar 数据驱动)+ `sidecar=Some(cmd)`;**勾「联网」** → 只设 `allow_network=true`(**`runtime_support` 永远留空**,见上 sidecar 坑⑤:写成 remote_daemon 会让 agent 被当非 PTY 拒→回退 shell)。诚实保持:无 sidecar 就真没用量环,不靠勾 cap 伪造。命令/Sidecar 字段 ASCII(`field != Name` 时 IME 关),名称字段中文 IME。**但 claude/codex 命令的 agent 用量自动出**(build_registry 走内置 adapter,见 sidecar 坑⑥),sidecar 字段只给其它 agent 的开发者,文案明示「不懂就留空」。
- **sidecar 遥测必须 per-pane 拉起,别塞进共享的 registry adapter**:`AgentRegistry.adapter(id)` 返回**共享** `Arc<dyn AgentAdapter>`(内置 Claude/Codex 是无状态日志解析器,共享 OK);但 sidecar 是**每会话一个子进程**,共享会让多 pane 抢同一进程 / 关一个 pane 杀掉别人的。**解法**:descriptor 只**携带命令**(`realtime_command`,来自 manifest `sidecar`),`TerminalView::new` 里**每 pane** `ExternalProcessAdapter::from_descriptor` 拉起自己的进程,存进 `realtime_adapter`(view 持有 → drop/`clear_agent` 杀子进程),事件 poller 跑在**这个 per-pane adapter** 上(不是 registry 的)。坑:① **spawn 在 pane 构造的非 unwinding 窗口**,`from_descriptor` 返 `Option<io::Result>`,失败只 `tracing::error` 降级"无遥测",**绝不 unwrap**(panic = 整进程 abort,见上 spawn 坑)。② **默认拒绝网络是单点决策** `AgentDescriptor::sidecar_launch()`(headless 可测):无 sidecar → `None`;`allow_network`(=`network_policy::Ask`)→ `Confirm` 弹卡;否则本地 stdio sidecar → `SpawnNow`。别把这逻辑散到 UI —— UI 只 match 三态。③ **只对 launched agent 起 sidecar**(launch intent),`sync_shell_agent` 敲命令切态**不**起 sidecar(hot path 不拉进程;诚实:shell 敲出来的 agent 无 sidecar 遥测)。④ poller 复用既有 `spawn_agent_event_poller`(120ms,agent 变即停),per-pane adapter 的 Arc 被 poller clone 持有 → `clear_agent` 后下一 tick 自然 break + drop → 子进程随之死。⑤ **血泪:sidecar 是否联网 ≠ agent 本体在哪运行。`sidecar_launch` 只看 `network_policy`(=`allow_network`),不看 `runtime_support`**。曾把 GUI「联网 sidecar」写成 agent 的 `runtime_support=["remote_daemon"]`(非 PTY)→ `from_profile` 的「不支持 LocalPty 就拒绝」守卫把这个 `claude` agent 当非 PTY 拒了 → **回退成 pwsh shell**(用户实测「怎么开成 shell」)。`claude` 命令显然在本地 PTY 跑;sidecar 联网只是个 `allow_network` 标志。编辑器对联网 sidecar **只设 `allow_network=true`,`runtime_support` 永远留空**(agent 保持 PTY)。⑥ **claude/codex 命令的配置 agent 自动用内置遥测**:`agent_host::build_registry` 对每个 manifest 先试 `tn_ai::builtin_adapter_for_manifest`(命令/别名命中 claude/codex → 内置日志解析器 + **用户自己的颜色/名称/Ink**,`usage` 强制开),否则 generic。所以用户加个命令为 `claude` 的 agent **自动有真实用量环、无需 sidecar**(这就是「用户重新注册 telemetry」的支持路径;平台 `tn-agent` 仍零知识,只 `tn-ai` 认名字)。`build_registry` 是 startup(`lib::run`)与 `reload_agents` 共用的唯一建表路径。
- **活动栏 git「变化即刷新」用 `notify` 递归监听,务必过滤噪声目录 + 防抖,且 git 与会话 mtime 解耦**:`spawn_change_watcher` 递归监听 agent pane 的 cwd → 变更触发 `refresh_changes`(后台有界 `git diff`)。**坑点**:① **`.git` 每次 git 操作都抖**(连我们自己跑的 diff 都触发)→ 不过滤会**自激刷新循环**;`target`/`node_modules`/`dist`/`.next` 巨大且与 `git diff` 无关 → 一并过滤(`is_noise_path`)。② 一次保存/构建触发**多文件事件** → **450ms 防抖** drain 成一次 diff。③ 监听器**存在 view 上,丢弃即停**(`change_watcher: Option<RecommendedWatcher>`;agent 退出/pane 关 → `clear_agent` 置 `None` → tx 析构 → 防抖任务的 `rx.next()` 收 `None` → 退出)。④ idle 时监听器阻塞在事件队列、**零唤醒**(保 idle 零开销)。⑤ git **从会话 mtime 门控解耦**(原绑 mtime 只在 agent 活动时刷)→ 现 agent 改文件、**用户手动改**都即时刷。notify 回调在自有线程 → `futures::mpsc::unbounded` 桥到前台防抖任务(同 reader wake 模式);drain 用 `try_recv().is_ok()`(`try_next` 已废弃)。
- **远端/WSL cwd 不能直接 `PathBuf::from` 当 Windows 路径**:`OSC 7 file://host/home/me` 解析出来的 `/home/me` 属于 Linux/macOS 命名空间,在 Windows 上直接 `PathBuf` 会误指向宿主盘根。`LaunchSpec::file_namespace` 是边界:Host 才能作为 Windows 进程 cwd;Workspace 只能把焦点 pane 转成该命名空间可浏览的 `ExplorerRoot` 后再驱动文件树。Host root 走 Windows 路径;WSL 启动追加 `--cd ~`,Explorer 用 `ExplorerRoot::Wsl` 把 Linux cwd 映射到 `\\wsl$\<distro>\...` 驱动 Linux FS;SSH/macOS 远端 cwd 走 `ExplorerRoot::Ssh` + `tn_pty::remote_fs::RemoteFileService`(SFTP 列目录 / Quick Look 有界预览 / 文本 guarded write-back),仍不得驱动本机 git watcher 或 Windows cwd。
- **SSH 远端 cwd 拿不到 → 文件树/「打开文件夹」永远停在 Windows 主机目录:`TerminalView::cwd()` 必须读 `BlockModel::cwd()`(模型级),不能只读 `current()`/`last_finished()` 的逐 block cwd**。SSH 注入的提示符脚本(`PtyEvent::Connected` 里)**只发** `OSC 633;P;Cwd`、**不含 A/B/C/D block 标记**(本地 pwsh 才发全套);`BlockModel::on_event(Cwd)` 把它存进**模型级 `self.cwd`**,但因为没有 block 标记**不建任何 block** → `current()`/`last_finished()` 恒 None。若 `cwd()` 只查这俩(+ `last_cwd`),SSH cwd 恒 None → `explorer_root_from_parts` 的 Ssh 分支 `cwd.filter(starts_with('/'))?` 返回 None → Explorer 不 re-root、「打开文件夹」也拿不到 root。**修:`cwd()` 兜底 `m.cwd()`(模型级,裸 `P;Cwd` 也能浮出)**;注入脚本钩好 `PROMPT_COMMAND`/`precmd` 后**立即调一次** `__tn_pc` 当下即报 cwd(不等下个提示符)。本地 pwsh/WSL(全套标记)不受影响。
- **远端目录 picker / WSL「打开文件夹」**:`open_folder_should_use_native_picker` 只对 **Host / 欢迎页**返回 true;**SSH 与 WSL 都走应用内导航 picker**(`RemoteDirPicker` + `PickerSource::{Ssh,Wsl}` + `PickerEntry`,SSH 经 SFTP、WSL 经 `\\wsl$\<distro>` 本地 `std::fs` 列目录)。cwd 未知时 `fallback_remote_root` 从 `/` 起,picker 必能开。picker 列表用 `uniform_list`(滚轮 + `↑↓` `scroll_to_item`),顶部固定「`..` 上级目录」行给鼠标上行路径。**键盘要活,见 gpui 焦点段「持自己焦点句柄的浮层」坑(进 `overlay_focused` + `disable_ime`)。**
- **reader 线程里 alacritty panic 会经 Mutex 中毒拖垮整个 app**:reader 持 `terminal.lock()` 时 `t.advance` panic → guard 在 unwind 中 drop → **Mutex 中毒** → 前台 render(GPUI 非 unwinding 回调)下一次 `.lock().unwrap()` panic → **进程 abort**(0xc0000409)。解法:在 reader 里**于持锁作用域内**用 `catch_unwind(AssertUnwindSafe(|| t.advance(...)))` 包住——栈只回退到 catch、guard 随后**正常析构**故 Mutex 不中毒,前台照常 lock(渲染半残状态总好过 abort);panic 后 break 停该 reader(grid 半改、不再喂)。外层再包一层 `catch_unwind` 把任何 reader panic 记 `tracing::error`(否则线程静默死、pane 无声冻结)。**别**改成"前台所有 `.lock()` 都 `unwrap_or_else(into_inner)`"(22 处、易漏)——持锁内 catch 一处搞定。

- **SSH 连接器现在只有两种来源要分清**:`最近/收藏`=Tn `ssh_recents.json`,成功连接自动记录、⭐收藏长期置顶、pen 只改昵称 name(可中文,不改真实 target)、再点星标取消收藏;`ssh-config`=OpenSSH `~/.ssh/config` 的 Host 别名,在 Tn 里**只读**。删除 ssh-config 源项必须让用户编辑 `~/.ssh/config`;连接必须走 alias,这样 `IdentityFile`/`HostName`/`Port` 仍由 OpenSSH 配置生效。普通 SSH 地址输入时禁 IME,收藏重命名态才启用 IME + `EntityInputHandler`;别把两条输入路径混在一起。

**gpui 渲染 / 材质**

- **gpui 0.2.2 on Windows**:可直接编译(DX11+DirectWrite,无需 Vulkan);首次编译数分钟。运行时 `HRESULT(0x887A002D)` 只是可选 DXGI _debug_ 层缺失——无害。
- **CJK / 宽字符终端渲染:跳 spacer + 每个 run 框「显式定宽到格」**:中文是**双宽字符**(占 2 格)——alacritty 存「字(WIDE_CHAR)+ 下一格 spacer(WIDE_CHAR_SPACER,c=' ')」。`row_runs` 若把 spacer 当普通空格渲染 → 每个汉字后多半角空隙(「间距那么大」);且网格按等宽 'm' 的 `cell_width` **flex 排版**,而 CJK 走回退字体(CaskaydiaCove Nerd Font **无 CJK 字形**)其字形步进 ≠ `cell_width` → 行内/行间漂移。**修**:① `row_runs` `continue` 掉 `WIDE_CHAR_SPACER`/`LEADING_WIDE_CHAR_SPACER`;② `CellRun` 加 `cols`(宽字符=2、余=1)= run 的真实**列跨度**;③ 渲染每个 run div **`.flex_none().w(cols×cell_width).overflow_hidden()`**——强制贴齐单元格网格,字形步进不符也只在 run 内轻微偏、run 边界必对齐。**别**依赖「等宽字体自然步进」假设排版(只对纯 ASCII 成立)。字形覆盖靠 DirectWrite 系统回退(用户能看到中文);要指定 CJK 字体可接 `[font].fallback`(gpui Windows `generate_font_fallbacks` 支持自定义回退链 + 叠系统回退,但无链式 API,需改 Style)。
- **反相块光标**:聚焦 = 实心块(光标色)+ **光标处字符以背景色重绘在块上**(锐利;别用半透明叠层=发糊);失焦 = `cola(cursor,0.55)` 细描边;IME 合成 `ime_marked` 内联(实心底 + accent 下划线)。**Quick Look 编辑器 caret 在 `uniform_list` 行内 inline**(`edit_row_cached` 光标处字符单独成 span 就地反色 → 无宽度位移;行尾用 `cursor_block`)。**固定单元格是 Quick Look 光标/选区精确的前提(血泪)**:按自然字流渲染时光标 x / hit-test / 选区全按 `disp_width×char_w`(CJK=2 列)算,而 CJK 字形步进 ≠ 2×char_w、误差随中文字数**累积** → 中文行光标漂、选区不跟手。修法同终端 `row_runs`:行改**固定单元格**(ASCII 串定宽 + CJK 单字 2 列定宽 + `overflow_hidden` 裁字形),列↔像素严格 = `disp_width×char_w`。**曾试 Q弹平滑滑动光标(绝对叠层 + 弹簧帧泵)已撤销**——帧泵高频重绘=卡、滑动半途字乱变。**教训:虚拟化列表给高频可变目标做帧泵动画 = 卡;编辑器光标要「瞬时准」不要「平滑漂」。**
- **长列表必须虚拟化,否则布局爆量冻结整窗**:把"几百行代码 × 每行逐 token 成 div"一次性塞进 `div().children()`,Taffy 会对全部元素跑布局(O(元素数)),每次重渲都重算 → 几百行就卡死整窗。**用 `uniform_list(id, count, |range, _, _| …)`**:只测首行高、**只渲可见 ~30 行**、其余惰性,带滚动(`.track_scroll(handle)`,handle 存 view 里每帧传)。闭包是 `'static`(签名给 `&mut App` 而非 `this`),要捕获的数据(行 Vec)用 `Rc` 零成本 clone 进去、行构建抽成自由函数。Quick Look 代码区即此(`quick_look.rs`:`file_row`/`diff_row` + `Rc<Vec<…>>`)。默认 `ListSizingBehavior::Auto` = 从 flex 父取高,放 `flex_1 + min_h(0)` 容器里即可。
- **别在 UI 线程同步 spawn 会阻塞的子进程(`git` 等),否则会冻死整窗**:Quick Look 曾在 `open()`(每次文件树点击 / 预览 `↑↓` 换文件都调)里**同步**跑 `git diff`(`.output()` 阻塞调用线程直到 git 退出)。真机实测冻结:gpui UI 线程被 git 卡住(大仓库 / `.git/index.lock` 被别的 git 进程占 / 杀软扫描 git.exe),整窗无响应。诊断关键:**冻结点在我没埋点的路径上**——日志在编辑态 `escape`(切回预览)后戛然而止、且**零** `SLOW` 告警(SLOW 计时只在编辑态分支),说明卡在预览键 → `Nav` → `open` → `git diff` 这条暗路。**修法(两层)**:① **惰性 diff**——`open()` 不再算 diff,只置 `diff_dirty=true`;只有用户**真切到 Diff tab** 时 `ensure_diff()` 才跑 git(`select_tab`/`save` 触发),把 git 从导航/打开/保存热路径**彻底移除**;② **有界 + 不闪窗**——`git_capture_bounded`:把 `.output()` 丢到一次性线程 + `mpsc::recv_timeout(1.5s)`,超时就放弃(孤儿线程/git 自行退出),`#[cfg(windows)] creation_flags(CREATE_NO_WINDOW)` 防控制台闪窗(同 `tn-pty::wsl`)。**别用 `spawn`+`try_wait` 轮询读 piped stdout**——大 diff 撑爆 64KB 管道缓冲会死锁(git 阻塞在写、我们等不到退出);`.output()` 在线程里把 stdout 读干净,无此问题。`explorer::compute_git_status`(`git status`)同加 `CREATE_NO_WINDOW`。
- **手写词法 scanner 必须每轮保证 `i` 前进,否则一个意外字符 = 死循环 → OOM 冻窗**:Quick Look 切到含 `①` 的 HTML 整窗冻(CLAUDE.md 无 `①` 故正常)。根因——`highlight()` 的标点分支 `while` 用 `d.is_alphanumeric()` 当终止条件,但词分支入口是 `is_alphabetic()`、数字分支入口是 `is_ascii_digit()`(都比 `is_alphanumeric()` 窄):`①`(U+2460,Unicode "No" 类)`is_alphanumeric()==true` 却 `is_alphabetic()==false`、`is_ascii_digit()==false` → 词/数字分支都不收,落到标点分支,而标点 `while` 在 j=i 处即 `break`(`①` 命中 `is_alphanumeric`)→ `j==i` 不动 → `i` 永不推进 → 无限 push 空 token → `out` 撑爆 → `rust_oom`(Windows 报 `0xc0000409`;交互式表现为 CPU 满、整窗冻、Ctrl+C)。**修**:标点分支末尾 `if j == i { j = i + 1 }`(吃掉这个字符当 1-char Plain),保证前进。**通则:任何 `while i<n { ... i = j }` scanner,每个分支都要证明 `j>i`;尤其当不同分支用 `is_alphabetic`/`is_ascii_digit`/`is_alphanumeric` 这类宽窄不一的判定时,留个"保底 +1"。** 加回归单测覆盖 `①②③½Ⅷ⑩㊀`。

WM_CHAR、不碰终端),非合成时经 WM_CHAR 回来(`backspace` 的 `0x08` 在 `replace_text` 里**重映射回 `0x7f`** 对齐 encode_key;编辑器则调 `backspace()` 删除)。**回车/Esc/方向键/Tab 仍编码**:方向键等无 WM_CHAR 回退(放行会废掉非合成态终端导航),回车/Esc 对终端太关键(LNM/vim 转义风险)——代价是合成期它们不进 IME(翻候选页用数字替代)。`on_key` 放行判定 = `!ctrl && !alt && !platform && (key.chars().count()==1 || key=="space" || key=="backspace")`。
- **IME 合成期的命名键(退格/回车/方向)被终端抢走 = gpui IMM32 之痛,根治靠 `VK_PROCESSKEY` 窗口子类(`platform::install_ime_keyfix`)**:上一条把放行集在 `space`/`backspace` 间反复横跳(b50f5fe 放行退格删拼音 → eda0560 又改回编码,因 gpui `parse_char_message` **过滤控制字符**[events.rs:1208 `filter(!is_control)`],退格 WM_CHAR 0x08 被吞 → 放行 = 键彻底丢失),本质是**打地鼠**:gpui 的 `is_composing` 只认 `GCS_COMPSTR`,而**微软拼音从不发**(`marked_text_range` 恒 None)→ gpui 从不短路键给 IME → 每键先进 `on_key`,我们只能**猜**该编码还是放行,合成期的退格/回车/方向必然猜错(被编码送了终端,IME 收不到 → 删不了拼音、回车直接执行命令、方向键不能翻候选页)。**这就是「输入法被限制」的真相,也是 WT 不会有此问题的原因——WT 走 TSF**(现代文本框架,IME 经 `ITfKeystrokeMgr` 在键到达应用前认领合成键),**gpui 0.2.2 走旧 IMM32**(且 `handle_key_event` 用 `ImmGetVirtualKey` 把 `VK_PROCESSKEY` 悄悄还原成真实键再派发,应用层分不清「合成中的退格」与「终端的退格」)。**根治**:在 gpui wndproc **之前**子类化窗口——凡 `WM_KEYDOWN` 且 `wParam==VK_PROCESSKEY`(0xE5,**OS 明示「此键属 IME」**)就替它 `TranslateMessage`(驱动 WM_IME_COMPOSITION → gpui `GCS_RESULTSTR` 提交链 → `replace_text` 写 PTY)并 `return 0` **消费掉**,gpui 永不会误编码它;IME **不要**的键以真实虚拟键到达(非 VK_PROCESSKEY)→ 原样透传给 gpui → `on_key` 照常编码。**IME 自己逐键决定要不要(按其合成态)**,我们只是honor → IME 无关、无需 fork gpui。重入安全(子类在 gpui wndproc 前、`TranslateMessage` 只投递不回调、VK_PROCESSKEY 直接 return 不进 gpui RefCell)。主窗(`workspace.rs` reveal 块)+ Quick Terminal(`ensure_init`)各装一次。`on_key` 的放行集(单字符+space)**保留**作非合成态英文 WM_CHAR 回退路径。诊断日志:命中时 info `ime subclass: routed VK_PROCESSKEY`,每键 raw vk 在 debug(`RUST_LOG=tn::ime=debug`)。**⚠ 真机待验**:此修依赖「微软拼音的合成键确以 VK_PROCESSKEY 投递」——理论成立(IMM32 标准),但须真机肉眼 + 看日志确认;若合成键不是 VK_PROCESSKEY(日志无 route 行、仍见 `on_key DEFER`),则该 IME 未用 IMM32 VK_PROCESSKEY 通路,需改走 gpui patch 或 ImmGetOpenStatus 启发式。
- **CJK 宽字符要拆成独立 run、别并入相邻 run**:`row_runs` 若把整段中文合并成一个 run,渲染时该 run 被强制定宽到 `列数×cell_width`,而 CJK 回退字体字形步进 < 2×cell_width → 多出宽度全堆 run 末尾 → **光标落到文字很远的右侧**。改为**每个宽字符自成一个 2 列 run**(`row_runs` 加 `last_wide` 守卫、宽字符不并入也不被并入)→ 逐格贴齐网格、光标紧跟文字;细微步进差摊成字间微距(要完全贴合需等宽 CJK 字体)。
- **不占分屏的浮层(Quick Look)要防「点裸终端偷焦点 / 事件穿透」——加 click-away scrim + 面板根吞点击**:Quick Look 浮在终端上、只占一块矩形(非全窗)。没有遮挡层时,点它**没盖住的**终端区域 → 终端 `glass_pane` 的 `on_mouse_down → focus_pane` 把焦点偷到底层 shell(编辑中尤其困扰)。解法:① workspace 在浮层下铺一层**正文区 click-away scrim**(`top 46 / bottom 30 / left=explorer 右缘`,**故意不盖文件树/标题栏/状态栏**——树仍能点开换预览、标题栏仍能拖),点 scrim = 干净关浮层(`ql_refocus` 还焦点给树/窗格),而非静默漂走;② 浮层**面板根** `on_mouse_down` 兜底 `stop_propagation`(点面板不冒泡到 scrim、不穿透到终端,且 `track_focus` 保持焦点在浮层);③ 浮层内可点子元素(代码行)各自 `stop_propagation`。**别用全窗 scrim**(会盖住文件树、断掉「点另一文件换预览」)。
- **聚焦 `track_focus` 浮层要在 `render` 里、别在动作回调里**:动作(如 `toggle_palette`)里 `handle.focus(window)` 时浮层**本帧还没渲染**,焦点落不上 → 键漏给底层焦点元素(实测:命令面板 ↑↓/Enter/Esc 全跑到底层终端)。解法:置 `*_needs_focus` 标志,在 `render` 里聚焦(Quick Terminal 启动器本就这样,所以一直正常)。
- **"哪个窗格被选中"要从 gpui 实际焦点反推,别只靠点击回调维护**:`tabs[active].focused` 原先只在点窗格外壳(`glass_pane` 的 `on_mouse_down → focus_pane`)时更新,但点进**终端正文**时 gpui 已把焦点切到该终端的 `track_focus`,`focused` 没跟上 → 焦点描边 + 「新会话」分屏基准都停在旧窗格。**解法**:在 `Workspace::render` 里把 `focused` **同步成真正 `is_focused(window)` 的那个叶子窗格**(遍历活动标签叶子);覆盖层/explorer 持焦点时无叶子命中 → 保留上一个,不乱跳。焦点是单一全局态,以 gpui 为准、render 里反推最稳。Quick Look 关闭后回**文件树**(`explorer.focus_handle()`,你是从列表打开的)而非某终端——别无脑 `refocus_active`。
- **跨实体键盘协作:让"当前活跃的那个实体"独占焦点 + 自己处理键,跨实体的事用事件回 workspace**(Quick Look 的模型,省去脆弱的焦点甩锅):浮层开着时 `QuickLook` 持焦点、`on_key_down` 包办预览导航 + 编辑;它办不到的(`↑↓` 要移**树**的选中、`Esc`/`Space` 关后焦点还给**终端**)发 `QuickLookEvent::{Nav,Close}` 给 workspace,workspace 调 `explorer::select_adjacent_file` / 关浮层 + `refocus_active`。入口由 explorer 自己的键盘 nav 引导(树 focus-on-click + `Space`/`Enter` 发 `OpenFile`)。两处焦点切换都走 `render` 里的标志(浮层自身 `needs_focus`;workspace `ql_refocus_pane`——事件回调没有 `window`,不能当场 focus)。打字插入读 `keystroke.key_char`(含 shift/符号/IME),特殊键读 `key`("backspace"/"left"/"escape"/"space"…);处理了就 `cx.stop_propagation()` 防冒泡重复(没处理的让它冒泡到 workspace,如 `Ctrl+Shift+J`)。
- **`uniform_list` 里渲染光标:行高必须固定**(`ROW_H`):`uniform_list` 只测**第一行**高度、其余按它排版,若光标行(带 caret 子元素)与普通文本行高度不同就会错位 → 给每行显式 `.h(px(ROW_H))`。caret 在虚拟化行内按 col 把行**切成 [前][caret][后]** 内联插入(等宽下不必测字宽);编辑缓冲的增删/光标全是 char 索引,落 `String` 前用 `char_to_byte` 转字节(多字节安全,headless 单测)。
- **`uniform_list` 鼠标点位:把 `on_mouse_down` 挂到「每一行」,行号 `i` 已知 → 免读滚动偏移**(`logical_scroll_top` 只在 `test-support` 下可读;但**纵向偏移本身生产可读**——`scroll.0.borrow().base_handle.offset().y` / `bounds_for_item(ix)`,见反相块光标坑)。每行闭包是 `'static`,捕获 `WeakEntity`(`cx.entity().downgrade()`)在事件里 `entity.update(app, |this, cx| …)` 改视图;列 = `(click.x − code区bounds.origin.x − 行号槽宽) / char_w`(`char_w` 用 `text_system().advance(font_id, size, 'm')` 量一次,code 区 canvas 捕获 bounds)。`Ctrl+S` 等写盘前注意:debug 跑的真机 `tn.exe` 会**锁住 `target\debug\tn.exe`**,headless 重链报 `os error 5 拒绝访问`——别 kill 用户在跑的实例,等它关了再 `cargo run`(`tn-ui` 库测 + 编译已能验逻辑)。
- **`uniform_list` 鼠标拖选 = 每行 `on_mouse_move`(行号已知)+ `pressed_button` 释放兜底 + 「含光标字符」语义**:拖选 `on_mouse_down` 置光标 + `edit_drag=true`;**每行**挂 `on_mouse_move`(左键按住)扩选到该行 `(i, col)` —— 同点位坑,行号 `i` 已知,绕开不可读的纵向 scroll offset。**释放清理**:`on_mouse_up` 只在 bounds 内触发,鼠标拖出浮层外松开收不到 → 行/根 `on_mouse_move` 检 `pressed_button != Left` 兜底清 `edit_drag`(同横向 thumb 粘连那条)。**「含光标字符」**:选区是半开 `[anchor, cursor)`,向右拖时光标(实心块)处那个字符不含 = 视觉「只选到块之前」→ move 改用鼠标悬停字符 `hover = floor(rel/char_w)`,相对锚点向右 `col = hover+1`、向左 `col = hover`,使选区始终**包含**鼠标所在字符。**预览态(只读)选区复用同一套**:`place_cursor`/`copy`/`select_all` 按 `editing ? buf : file_data 的 lines` 取内容源(预览态 `buf` 是空的),选区触及行用 `edit_row_cached`(caret 传 `(usize::MAX, usize::MAX)` 永不命中 = 不画光标)显选区底色,`on_key` 预览分支放行 `Ctrl+C`/`Ctrl+A`、其余控制键仍忽略(不可改)。
- **`WindowKind::PopUp` 窗口里 keymap/action 不派发**:`key_context`+`on_action` 绑的 `Ctrl+Shift+L`/`Ctrl+Tab` 在 quick 窗口**都无反应**(原始 key_down 能到焦点终端、能打字,但 binding 不匹配);主窗口(Normal)经 SendInput 验证可派发。**结论**:别依赖 PopUp 窗口内的 keymap 动作;quick 窗口"换 agent"改用**退出会话→`ProcessExited`→回启动器**(走实体事件,可靠)。
- **(M5)外部 `SetWindowPos`/`ShowWindow` 不能在 gpui 更新回调里同步调**:它们**同步**把 `WM_SIZE` 派回 gpui 窗口过程并 `borrow_mut` 窗口状态;若此时正处在 `window.update` / `observe_window_activation` / `Context` 回调里(已持该 `RefCell`),就**重入借用** → resize 被静默丢弃("RefCell already borrowed"),窗口停在旧尺寸。**解法**:所有窗口操作(`make_topmost`/`set_bounds`/`show`)丢进 `cx.spawn` 前台任务(借用释放后),取焦放 `render`。诊断:reveal 时打 `scale` + `GetMonitorInfoW` 工作区 + 算出的 shown 矩形——几何对却不生效 = 重入借用,非 DPI。
- **失焦后唤不出命令面板 / 同时不破坏拖窗 = 全窗焦点锚点(根)+ 拖拽条 `.occlude()` + `on_focus_out` 重锚(三层,缺一不可)**:gpui 按**焦点**派发键绑定,动作上下文 `key_context("Workspace")` 挂在根上,所以焦点一旦掉成 `None`(点状态栏/chrome 空白、关掉浮层、程序性 blur),`Ctrl+Shift+P` 等就**派发不出**。① **整窗锚点**:`track_focus(workspace_focus)` 放**窗口根**——点哪儿都能把焦点收回 Workspace 上下文(曾放 `body` 只盖工作区,漏掉状态栏/标题栏空白/关浮层,**实测不够**)。② **但** `track_focus` 元素带「点击即聚焦」监听、命中调 `window.prevent_default()`(div.rs:2025),而 Windows 拖窗经 `window_control_area(Drag)` 的 **NC 命中**;gpui 在 `handle_nc_mouse_down_msg` **先把按下派发进元素树**,`!propagate || default_prevented` 为真即 `return Some(0)` 吞掉系统拖窗(DefWindowProc 永不触发)→ 全窗根锚点会**静默拖不动**。解法:给标题栏**拖拽条 `.occlude()`**(BlockMouse)——`hit_test` 命中 BlockMouse 即 `break`,根 hitbox 不进 hover 集 → 根的点击聚焦**不触发**(拖拽存活),而 `on_hit_test_window_control` 用 `ids.contains` 仍认它(HTCAPTION)。两不耽误。③ **非点击失焦**(关浮层 / 程序性 blur,没有点击去触发②的重锚)靠 `window.on_focus_out(&workspace_focus, …)`:`workspace_focus` 在根上、子树=整棵树,故 focus_out **精确在「焦点离开一切(→None)」时触发** → 回调里 `workspace_focus.focus(window)` 立即重锚(只在 `is_window_active()` 时,免 Alt-Tab 被拽回;重聚焦是 focus-**in**、不循环)。render 里的泊车留作「窗口重新激活」兜底。**浮层(命令面板/新会话/布局)聚焦也从「开时 `*_needs_focus` 一次性」改成「开着且当前未聚焦,每帧重抢」**(幂等 `focus()` 已聚焦即 return,不循环)——一次性有时首帧没落上,键漏给底层 shell(它的字段注释自己写了 "doesn't reliably land")。
- **持自己焦点句柄的浮层必须同时进 render 的「reveal 聚焦块」和「焦点反射块的 `overlay_focused` 排除项」——只进前者会被后者每帧抢走(血泪:远端目录 picker 键盘全死的真凶)**:`Workspace::render` 末段有一段「从 gpui 实际焦点反推当前选中 pane」(把 `tabs[active].focused` 同步成真正持焦点的叶子,见上条),它 gate 在 `overlay_focused`(开着的浮层列表)——**只在没有浮层时才该跑**。新加的浮层(`remote_dir_picker`)若**漏进这个列表**:浮层开着时该块照跑 → 发现「没有 pane 持焦点」(焦点其实在浮层的 `track_focus` 上)→ 走 else 兜底 `workspace_focus.focus(window)` **每帧把焦点重锚到根** → 正好抢走 reveal 块刚给浮层的焦点 → 浮层 `track_focus` 永不持焦 → `on_key_down` 永不触发 → **键盘全死**。命令面板/分屏/布局/ssh_prompt/agent_form 之所以好,**正因它们在 `overlay_focused` 里、这段被跳过**。**通则:加任何新浮层(自带 focus handle + on_key_down 导航),要在 `render` 里改两处——(a) reveal 聚焦块加 `if open { focus }`,(b) `overlay_focused` 加 `|| self.<overlay>_open`。漏 (b) = 焦点被每帧抢走,表现为「鼠标能用、键盘全死」。** 另:这类**无 `EntityInputHandler` 的纯导航浮层还要进 `disable_ime`**(picker/split/layout/palette)——否则活动 CJK IME 把 `↑↓/Enter/Esc` 当 `VK_PROCESSKEY`,被 `install_ime_keyfix` 子类 `return 0` 吞掉(见 IME 子类坑),键同样到不了 `on_key_down`。两道一起才保导航键活。
- **`Ctrl+Shift+*` 在中文/多布局 Windows 可能被 IME/布局切换吞掉**(系统热键也是 `Ctrl+Shift`)。已(SendInput)验证绑定/派发正确,但真机键入可能不达 app:可改键避开,或把系统"输入语言热键"设未分配。`Ctrl+Tab` 与鼠标点击聚焦不受影响。侧栏 explorer 走 `Ctrl+Shift+B` 开关、Quick Look 速览浮层走 `Ctrl+Shift+J` 收起(浮层不抢焦点 → 动作在 Workspace 层稳派发;再点另一文件即重定向;真机实测 `Ctrl+Shift+B/J` 可用,IME 吞键时可改键)。
